## 1. 実現可能性の総合評価

### 1.1 意図は理解できるか — 完全に理解できる

狙いは設計から明確に読める:「同種の質問への再推論の無駄を共有意味キャッシュで削る/MeanCache が放棄した"共有の信頼性"を、事実型限定・独立検証・非匿名の説明可能性で埋める」。競合分析で新規性を「ユーザー横断共有の信頼性」一点に絞れているのは正確な自己認識。文書の規律(非目標明示・誤分類コストの非対称性・脅威対応表・裏取り付きRoadmap)は構想文書として上位数%の品質。問題は文書の質ではなく前提の一部にある。

### 1.2 面白い点(本物の強み)

- 投票を「真偽の決定」から「優先度ヒント」へ格下げしたシビル対策(信頼性設計メモ §8.1 / Architecture §8.1)は筋が良い。
- 誤分類コストの非対称性(volatile→permanent 誤りだけが致命的 → 昇格厳格・降格緩和)を一貫。
- トリプルのみ保存・受信側再合成という著作権距離の取り方は独創的、法的分析(信頼性設計メモ §8)は素人水準を超える。
- PoC の進め方(不変条件の明文化、ゲート制、S2 完了時に自ら残存リスク3件を掘り出し S3 前提化)は模範的。

### 1.3 致命的に無理がある点(構造的問題、パッチでは直らない)

1. **共有できるもの ≠ 共有する価値のあるもの**。安全のため共有対象を「恒久・事実」に絞った結果、それは最も安く手に入る知識(モデルの重み内/Wikidata/検索)になり、推論が本当に高価な質問(コード・長文脈・推論・個人文脈)は全部ゲートで弾かれる。節約したいコストの発生場所とキャッシュが効く場所が一致していない。自前 S2 実測でも共有可は 4/12(33%)、読み取り側ヒット率はさらに「他人が過去に τ≥0.9 の同一質問をした確率」を掛ける。経済性が成立するヒット率シナリオが描けていない。
2. **検証コストが節約分を食う**。独立生成・層3抜き打ち再推論・slow TTL 再評価で、ネットワーク全体ではむしろ複数回推論する収支。全体の推論収支モデルがどこにもない。
3. **「独立検証」が独立でない**。主要 LLM は学習コーパスを共有していて同じ誤りに独立到達する。トリプル一致率が測るのは"真実"でなく"学習データの合意"。自然発生キャッシュ毒(共通原因故障)に無防備。
4. **インセンティブの内部矛盾**。法的防御のため匿名性(=分散共有型ネットワークの参加エンジン)を捨てた上で人類規模を前提。フリーライドが支配戦略、評判で何が買えるか未定義。

### 1.4 見落とし・過小評価リスク

- 名誉毀損・個人情報/GDPR の軸が §11 に欠落(「事実トリプル」は著作権には効くが名誉毀損の抗弁にならない。L0 の個人参照フィルタは一人称「私の」しか見ておらず、第三者の個人データを弾く設計がない)。
- ~~「Winny」という名前自体がR6(幇助論回避)と矛盾する法的負債~~ → 2026-07 のリブランド作業(ブランドから「Winny型」表記を除去)で対応済み。
- regurgitation フィルタは参照コーパスを各ノードに配れないため分散環境で原理的に実装困難の可能性。R2 の柱が揺らぐ。
- revocation の遵守が任意なプロトコルが法的に「止められる仕組み」と認められるかは疑問。
- 多言語分断(「Winnyとは?」と "What is Winny?" は τ≥0.9 で当たらない)でヒット率がさらに割れる。
- 上流での問題消滅リスク(推論単価下落・端末内 SLM 普及)が P2P ブートストラップより速い。

### 1.5 総合見立てと出口3案

技術的部品(S3〜S5)は作れる。だが構想全体(Public層・人類規模)の実現性は低い。理由は防御が破られるからではなく(むしろ平均以上に固い)、守っているものの価値が細いから。

- **出口1: Company層を主戦場に**(社内ならシビル無し・法務激減・ヒット率現実的・インセンティブは業務命令で解決。既存資産がほぼ活きる。製品として成立しうるのはここ)。
- **出口2: 看板を掛け替える** —「出所署名・失効可能・分散検証つき事実知識グラフ」(分散版 Wikidata+信頼層)。
- **出口3: 続けるなら最低限** — 名前からブランド性を外す(→2026-07リブランドで対応済み)/名誉毀損・個人情報軸を R8 追加/witness 自体の信頼モデルを閉じる/推論収支が黒字になる条件を1枚のモデルにする。S3(P2P化)は工数最大なのでこの収支モデルが立つ前に進むのは投資順序の誤り。

## 2. エントリ形式・ハッシュの弱点と改良案(cache.rs / signer.rs を読んだ上で)

### 2.1 核心

今のハッシュファイルは「内容アドレスストア」と「可変な判定状態の置き場」が1ファイルに同居した中間物で、ハッシュが守っている範囲と守っていない範囲の境界が形式に現れていない。`entry_id = sha256(signed_payload)` がファイル名なのに、embedding/volatility_confidence/evidence/shareable/agent はハッシュにも署名にも入っていない(「ファイル名=内容の一部のハッシュ」)。

### 2.2 弱点(重要度順)

1. **未署名フィールド経由の改竄が verify() を素通りする(最重要)**。特に embedding はロード時に再計算されず検索にそのまま使われる(cache.rs:261-268 は hash 再計算+author_sig のみ)。ファイルの embedding だけ書き換えれば署名もハッシュも有効なまま無関係な人気質問にヒットさせられる。S3 で他ノード由来エントリが入った瞬間、署名検証を無傷で通過するキャッシュ毒の主経路になる。shareable/agent も同様。(Roadmap の S2 脅威レビューで自己検出済み=評価できるが、「S3 の課題」でなくハッシュ設計そのものの構造欠陥)。
2. **正準化が serde_json の内部実装依存で脆い**。BTreeMap キーソートは仕様保証の正準形でない。(a) feature unification で `preserve_order` が推移的依存に有効化されたら全既存エントリがハッシュ不一致で無効化、(b) C#(Blazor)/他ノードとバイト一致でハッシュ再現する保証がない、(c) Unicode 正規化なし(Architecture §6 の「正規化質問」が未実装)。「禁止をドキュメントで守る設計は負け」。
3. **可変であるべき volatility class が不変な内容アドレスに封じ込められている**。Architecture §10 は class の昇格・降格が正常動作なのに、class が signed_payload に入るため再分類で entry_id が変わり署名無効化、witness/trust/失効通知が迷子になる。confidence/evidence を署名から外した判断は矛盾に途中まで気づいた証拠だが class 自体が同じ理由で外に出るべき(あるいはタグ更新を別の署名付きオブジェクトに)。Architecture §6 が class を署名対象に含めている点で設計側にも問題が波及。
4. **created をハッシュに含めるため同一 Q&A が無限に重複エントリ化**。witness・独立検証・信頼度が複製間に分散して薄まる。攻撃者は created だけ変えて量産可能。しかも `Local::now().format(...)` はタイムゾーンなしローカル時刻で ISO8601 不完全、claimed_date はノード間比較値なので UTC/RFC3339 でないのは将来バグ確定。
5. **Architecture §6 と PoC で entry_id 定義が食い違う**(Architecture §6=hash(正規化質問群+Agent+Model)=質問スロットID/PoC=sha256(question+answer+created+volatility+facts)=回答インスタンス ID)。Architecture §6 版は §5.2 の「複数回答版併存」とも矛盾。設計書内部でも未確定 → オーナーの「微妙」は正確。
6. (小)スキーマ進化とドメイン分離がない。payload にバージョン識別子なし(Architecture §6 schema_ver 未実装)、署名にドメイン分離タグなし(将来 witness/revocation/投票署名の cross-protocol 流用を防げない)、DummySigner の `sha256(secret||payload)` は長さ拡張攻撃が成立(HMAC にすべき)。

### 2.3 改良案(S3 着手前が唯一のタイミング。以後は形式変更=全ネット非互換)

- **A(最重要): 署名対象バイト列をそのまま保存する方式へ転換**。構造体→再シリアライズ→照合をやめ、正準ペイロードのバイト列自体を不変部として保存し、verify はそのバイト列を直接ハッシュ・署名検証、利用時にパース。serde_json 再現性問題(弱点2)が原理ごと消える。
- **B: 不変コアと可変状態の物理分離**。`<id>.entry`(署名済み不変コア=schema_ver+正規化question+facts+provenance+created+初期 volatility、id はこれのハッシュ、以後書き換えない)と `<id>.state`(confidence/evidence/shareable/局所 trust/再評価履歴=各ノードが自分で再導出する値、署名しない)。class 更新は state 側に「署名付き再評価レコード」を追記するイベントソーシング型 → Architecture §10 と保存形式が初めて一致(弱点3解消)。
- **C: embedding をファイルから消す**。ロード時に question(署名済み)から再計算。加えて embedder のモデル/バージョン識別子を不変コアに記録(将来モデル交換時にベクトル空間が混ざる事故防止。現行 Roadmap 未記載の追加指摘)。
- **D: ID の二層化**。`question_key = hash(正規化質問クラスタ)`(検索・重複排除・バージョン束ね)と `entry_id = hash(不変コア)`(回答インスタンスの内容アドレス)を分離。created は entry_id に含めてよいが重複排除は question_key 側で。Architecture §6 の entry_id 定義もこの二層に改訂(弱点5解消)。
- **E: 形式の衛生**。RFC 8785(JCS)準拠か長さ接頭辞付きバイナリに正準形を仕様化/payload 先頭にドメイン分離タグ+バージョン(例 `nyllm/entry/v1\n`)/created は UTC RFC3339/質問は NFC 正規化。

補足: オーナーの「(テキストでなく)推論計算時の結果の方が良いのでは」という直感の正しい実装形は、(A)=正準化された構造化事実(facts/トリプル)を第一級の保存対象にしテキスト回答を付随データに格下げする方向。(B)=埋め込みベクトルやロジットそのものをハッシュ鍵にするのは原則 NG(浮動小数点の非決定性で正準化が破綻)。

→ 本改良案A〜Eを具体化した確定設計 = [S2.5_エントリ形式設計.md](./S2.5_エントリ形式設計.md)。

## 3. セキュリティ脅威モデル(敵対的棚卸し)

### 3.1 前提: 2フェーズ

- 今の PoC: 単一ノード・単一鍵・ローカルファイル。攻撃者=そのマシンにファイルを書ける者=OS の問題。DummySigner は自鍵 MAC で他者検証不能。意味のある対外的攻撃面はまだ存在しない。
- S3 以降: エントリが他人由来・鍵が複数・配送が P2P になった瞬間に全攻撃面が一斉に有効化。セキュリティ設計 ≈ S3 の設計。今がそれを固める唯一のタイミング。

### 3.2 全リスクが収束する3点

**「鍵は安い・誰でも witness になれる・LLM の独立は幻」**。信頼性設計メモ §6 のシビル設計は悪意ある多数決攻撃には平均以上に考え抜かれているが、この3点は章立てすら弱い。

### 3.3 攻撃面の棚卸し(要素→信頼前提→崩れる条件)

- entry_id/署名: 署名対象が内容全体を覆う前提 → embedding/shareable/agent/confidence が署名外(§2.2 弱点1)。
- witness(誕生証明+時刻): 独立第三者で多数派が正直 → witness 資格に制約なし=自作自演で全 witness を占有可能。
- 評判/trust 集計: 検証済み貢献が真に検証済み・評判は移転不能 → 独立検証が非独立、鍵は生成コストほぼゼロで移転自由。
- 投票(優先度ヒント): 毒されても致命的でない前提 → ヒントでも探索順序=露出=消費を支配。
- 揮発性タグ更新: 昇格厳格・降格緩やか・反証に検証済み事実要 → 検証済みの生成源がシビル可能なら両方向に操れる。
- 共有ゲート: L0 語彙+L2 自己申告+トリプル分解で毒/文脈依存を落とす → L0 は部分文字列マッチ約40語+述語オントロジー約20語(`volatility.rs:27-40`)、回避容易かつ受信側が再判定できない(shareable 未署名)。
- 独立検証(トリプル一致): 別 Agent/別時期の一致=真実に近い → モデル間コーパス共有で一致=学習データの合意。共通原因故障に無防備。
- DHT/P2P 配送: ルーティング正直・キー空間担当は公平 → Kademlia 系は Sybil+eclipse の古典的標的。
- 失効(revocation): 通知は伝播し各ノードが自律的に従う → 遵守が任意、権限モデルなし=握りつぶしと濫用が両立。
- ブートストラップ: 最初に繋ぐピア/紹介者が善意 → 初期ピアリストを攻撃者が握れば世界観を注入(eclipse の入口)。

### 3.4 攻撃シナリオ(優先度付き)

- **【1・最重要】witness 自作自演 + eclipse による毒の恒久化**。複数鍵生成(ID 生成 PoW「数分」は並列化で無力化)→ 毒エントリに自分の別ノード群から「最初に受信した」witness 署名を大量付与し独立多数に偽装 → DHT で標的質問のキー近傍を攻撃者ノードで固め(eclipse)被害者の検索を攻撃者視界に閉じ込め → 高信頼ラベル付きで毒を恒久配信。Architecture §10.3 攻撃A(volatile→permanent 偽装)も時系列を自作できるので成立。現設計の防御は効かない(witness 自身の信頼を担保する機構がない)。**本システムの単一障害点**。
- **【2】自然発生キャッシュ毒(モデル間共通原因故障)**。攻撃不要。Web 上の誤情報を各 LLM が独立に同じ誤りで再現 → 一致率高 → 高信頼の毒として昇格・恒久化。層3抜き打ちも別 LLM に聞くだけで同じ相関の内側。信頼性設計メモ §6 はシビル(悪意)には手厚いが共通原因故障(悪意なし)には章立てすら存在しない。事実型限定の根幹を揺るがす。
- **【3】検閲失効と失効の握りつぶし(両刃)**。失効に権限モデルなし・遵守が任意 → 正当エントリへの失効フラッディング(検閲・削除)、自分の毒への失効を eclipse 近傍で握りつぶし、の両方が成立。Architecture §10.3 の「反証必須・複数版併存」は降格には効くが失効には明示されていない。失効者の資格が未設計。
- **【4】whitewashing + 評判ブートストラップ攻撃**。鍵生成が安いのでスラッシング後は鍵を捨て再参加。「時間は並列化不可」は正しいが 1回の出し逃げ・使い捨て前提の攻撃者・紹介制度悪用(擬似社会グラフ捏造)には無力。「評判で何が買えるか」未定義なので攻撃の旨味自体が不明(=正当な貢献動機もない、インセンティブ矛盾と表裏)。
- **【5】プライバシー — クエリメタデータと witness による行動追跡**。R4 で「説明可能性=非匿名」を売りにした法的判断が、プライバシー上は自らトレーサビリティを最大化。DHT lookup は近傍ノードにクエリ(=何を聞いたか)を晒し、witness 署名の時系列は特定ノードの活動タイムラインそのもの。Public 層内部のクエリプライバシーはゼロ。
- (補足・確実)共有ゲートの回避。L0 は固定語彙のため語彙表を避けた言い回しで時事情報を permanent 偽装可能。さらに shareable が署名対象外で受信側再判定できないため、S3 では「shareable を信頼せず受信側で必ず再導出」が絶対条件。

### 3.5 witness の循環的信頼は解けるのか

結論: permissionless(誰でも参加)を保つ限り完全には解けない。緩和はできるが必ず「時間」か「外部アンカー」のコストを払う。構造は Sybil 耐性の一般不可能性(Douceur 2002、Architecture §2.2 N4 が引用)そのもの(誕生証明を発行するのが witness で、その witness の独立性を保証するのが誕生証明 → 循環)。緩和策(いずれも「解決」でなく「値上げ」):

1. **外部時刻アンカー(最も現実的)** — エントリ ID 群の Merkle ルートを公開 append-only ログ/RFC3161 TSA/Bitcoin ブロックハッシュ等に定期アンカー。優先度1の「時系列捏造」を直接封じる。免責(Architecture §11)にも効く一石二鳥。信頼性設計メモが RFC3161 を「中央依存」で退けた判断は再考に値する。
2. witness 資格の希少化(循環を評判層に押し込むだけ、単独では不十分)。
3. k-独立性の物理検証(IP/AS 多様性、VPS/ボットネットで回避可能、気休め)。
4. DHT eclipse 対策(S/Kademlia、ノード ID を IP 等に束縛、複数の素なルーティング経路)。優先度1の eclipse 部分に必須。

正直な結論: 外部時刻アンカー(1)+DHT eclipse 対策(4)で「攻撃を割に合わなくする」ところまでは行ける。だが「純粋 permissionless + 中央要素ゼロ + 完全 Sybil 耐性」は同時に成り立たない(トリレンマ)。設計はどれを諦めるかを明示的に決めるべきで、現状は「中央ゼロ」を暗黙に最優先しその代償を「国家級 APT は排除不能」の一行(Architecture §6)で流している。

### 3.6 S3 着手前に閉じるべきトップ5(対策セット)

1. **witness の循環を断つ外部時刻アンカーの方針決定** — アンカーの有無でエントリ形式(Merkle 化・アンカー参照フィールド)が変わる。信頼性設計メモの RFC3161 不採用判断を「免責と過去詐称防止のための限定的中央アンカーは許容」に改訂。完全分散に拘るなら公開ログ(Certificate Transparency 型)。方針だけでも今決める。
2. **受信側での共有可否・内容の完全再判定を不変条件化**(shareable/embedding/agent の署名 or 再導出)。§2.2 弱点1・Roadmap 残存リスク a/b/c そのもの。改良案A(§2.3、署名対象バイト列保存)と束ねて実施。「送信者の shareable 主張は一切信頼しない」をプロトコル不変条件に明文化。
3. **DHT eclipse 耐性の設計選定** — eclipse は優先度1・3・5 攻撃すべての入口。ノード ID=hash(公開鍵)に PoW 束縛+IP/AS 多様性制約、キー近傍 lookup を複数の素な経路で冗長化、ブートストラップピアの多様化。後付け不可。
4. **共通原因故障(モデル間相関)への「独立性」定義の修正** — independent_agreement にモデル系統の多様性を要求(同一ベースモデル/同一プロバイダの一致は割り引く)、外部権威ソース(構造化知識ベース/一次資料 URL)照合を独立検証の一項目に追加、「LLM の合意=真実」を弱める。Architecture §8.1 に但し書き。
5. **revocation の権限モデルと遵守モデルの定義** — 失効種別を分離((a)著作権/違法系=発行者資格限定+遵守強め/(b)誤情報系=削除でなく Architecture §10.3 降格+反証必須)、失効伝播に重複排除と再フラッディング(握りつぶし対策)、失効自体にも評判/反証の縛り。「誰でも何でも失効できる」を最初から禁じる。

### 3.7 断り書き

本評価は設計・PoC コードに基づく技術的分析であり法務判断ではない。R3/R4/プライバシーの法的当否はドキュメント記載どおり専門弁護士レビューが別途必要。

> **本節も§1〜§3と同様、レビュー指摘・検討事項の記録であり採用済み設計ではない。** Architecture.md / 信頼性設計メモ.md 本体には今回反映していない(冒頭の断り書き参照)。プロジェクトオーナーとの追加対話ログ(2026-07-17実施)を、既存本文のトーンに合わせて整理したもの。actionable な項目のみ [Roadmap.md](./Roadmap.md) から索引する。

オーナーが連続して提示した4つの直感が、いずれも**「検証コスト」という一本の軸に収束する**という発見の記録。毒(キャッシュ・ポイズン)の害とキャッシュの価値は同じコインの裏表であり、「検証できない答えを他人から受け取る」という同一行為の光と影である、という認識が全体を貫く。

**全体を貫く対称性(何度も再出現する)**: 毒が無害な領域(低リスク・検証容易)= キャッシュも無価値な領域(ローカルSLMで足りる恒久事実)。キャッシュが高価値な領域(検証が高価)= 毒が致命的な領域。安全弁(「おかしければ再推論する」)が効く所は価値が薄く、価値がある所は安全弁が効かない。片方だけを都合よく取ることはできない。

### 4.1 直感①「ハッシュファイルの内容の扱いが微妙」→ エントリ形式再設計(§2 に集約済み)

既存本文§2(弱点1〜6・改良案A〜E)にすでに集約されている論点であり、ここでは再掲しない。オーナーの「(テキストでなく)推論計算時の結果の方が良いのでは」という直感の正しい実装形は、**改良案A**(正準化された構造化事実=facts/トリプルを第一級の保存対象にし、テキスト回答を付随データへ格下げする方向)である。埋め込みベクトルやロジットそのものをハッシュ鍵にするのは原則 NG(浮動小数点の非決定性で正準化が破綻するため)。

→ 詳細は §2.3 改良案A・B・D を参照。

### 4.2 直感②「ハッシュをビットコイン的に(改竄不能な時刻付き台帳)」→ 時刻/順序アンカー二層

既存§3.6 課題1(外部時刻アンカーの方針決定)・§3.5(witnessの循環的信頼)で「緩和策はあるが解決ではない」と結論づけた論点を、具体的な二層構成にまで踏み込んだもの。本節は具体案としての位置づけであり、循環そのものの評価は§3.5に譲る。

**アンカーが提供できるのは本質的にただ一つ、「このデータは時刻T以前に存在した」という証明(existence-before-T)である。**

- **封じられる**: 過去日詐称(Architecture §8.3「時間は並列化不可」に外部の裏付けを与える)。witness循環のうち"時系列捏造"の一辺。
- **封じられない**: Sybilの存在そのもの(若さ偽装を防ぐだけ)、eclipse(配送経路の問題。→ §3.6 課題3)、non-existence証明。

**候補比較**:

- **(a) 内蔵追記専用連鎖ログ(CT型・gossip)**: 循環を閉じず値上げ止まり(split-viewに弱い)。中央要素ゼロ・最良スケール・eclipse耐性(G1)最良・法的強度は弱い。
- **(b) 外部TSA(RFC3161)**: 時系列は最強に閉じる。ただし**中央要素を全放棄**(単一障害/課金/検閲点になる)。スケールは悪く、法的強度は最強。
- **(c) パブリックチェーン定期アンカー(OpenTimestamps型、Merkleルートのみ刻む)**: 時系列を非中央のまま実用上閉じる。中央要素はほぼゼロ(公共インフラで管理者なし)、スケールは良好(数百万件→ルート1個)、eclipse耐性(G1)は良好(既存PoWに相乗りするだけで自分では合意もPoWも走らせない)、法的強度は中〜強。
- **不採用**: グローバルNakamoto合意やエントリ毎PoW — 非目標N3(全順序合意)と正面衝突し、かつ本システムは全順序を要求しない(結果整合+複数版併存で足りる)。

**推奨: (a)+(c)の二層。** (a)=即時・高スループット層、(c)=低頻度の"真実の錨"層(推奨アンカー間隔は概ね1時間 — §8.3が要求する解像度は日〜月なので過剰なほど十分)。(c)を(a)に被せると、ローカルログのhead hashが公開アンカーと一致せねばならなくなり、**split-view攻撃が「公開アンカーとの矛盾」として検出可能**になって(a)単体の弱点が相殺される。

**S3設計スケッチ**(§2.3 改良案A・B・Dと統合する前提):

- Merkleリーフ = 署名済み不変コアのハッシュ = `entry_id`(=core_hash)。可変状態(confidence/shareable/trust)はアンカーしない。
- アンカー参照は非同期の可変添付フィールド `timestamp_proof`(登録直後は `pending`、次アンカー間隔で `merkle_path` + `anchor{block_height, txid, root}` を充填)。署名対象にも `entry_id` にも含めない。
- 受信側検証は4点: ①内容(core_hash+author_sig)②shareable/embeddingを信頼せず再導出(→ §3.6 課題2と同じ要件)③時刻(Merkle包含証明→ブロック時刻でcreatedの過去/未来詐称を両方棄却)④一貫性(送信元ログheadが公開アンカーと矛盾しないか=split-view検出)。
- 「アンカー済みか否か」で信頼度を2段階化する。

**残る穴**: eclipseは残る(→ §3.6 課題3が別途必須)/witnessの"独立性"の辺は閉じない(時刻Tに既にSybilが揃っていればアンカーは正直にそれを証明するだけ)/共通原因故障(→ §3.4【2】・§3.6 課題4)には無関係/未アンカー窓(推奨1時間)の出し逃げ/外部インフラ依存(緩和策=アンカー先の差し替え可能性+複数チェーン併行)/アンカー役のサボタージュ均衡。

### 4.3 直感③「情報=採掘(PoUW)」→ 袋小路。ただし部品だけ拾える

**判定: 「情報=採掘」そのものは袋小路。** ただしその失敗の中に部分的に救える最小構成が残る。

袋小路である理由:

- 生の推論はPoWの4条件(決定的・一意・検証非対称・仕事と成果の束縛)を1つも満たさない。古典PoWがすでに欠く2性質(検証の非対称性/識別子束縛)に加え、対話で新たに3つ判明した: 出力の非決定性(temperature>0で検証が意味判定に化ける)、モデル非同一性(「同じ仕事」が定義できない)、プロンプト自由度(毒の答えに合うプロンプトを後から作れる=仕事の方向が攻撃者有利)。
- **符号が逆(最重要)**: 正直者はフルコストで正しい答えを出すが、毒攻撃者は推論せず嘘を書けばゼロコスト。PoUWは「正しい情報を作る側にだけ課金し、毒を作る側は課金回避」してしまう。出力をコピーすれば推論コストはSybil数で割られ実質ゼロに漸近する。
- 別建ての古典PoW(コンテンツと無関係な計算課題)でも符号逆は解けない — 毒も正直も同じPoWを払うだけで、量制限にしかならず"1個の高信頼毒"には効かない。
- 真の障害: **「内容の真偽」は時間と違って外部にコミットできない。** 真偽を刻める公開チェーンは存在しない(あれば独立検証もLLMも不要になる)。ゆえに真偽の循環はwitness循環より本質的に固く、アンカー(→ §4.2)で閉じた手が使えない。

2026年時点のzkML見立て: フルzkMLは非現実的(証明生成コストが推論本体の数千〜数万倍)。今使える部分形態はモデルコミットメント+サンプリング検証・決定的推論の再現・TEE(ただしハード製造元への中央信頼が発生)。**いずれも「推論をサボったSybil」には効くが「毒(内容の真偽)」には効かない。**

**救うなら最小構成**(「情報=採掘」の夢は捨て部品だけ拾う):

1. モデルコミットメント — 推論サボりSybilを排除、安価に今できる。不変コアのprovenanceに `model_hash` を追加。
2. 評判ステーク+チャレンジ報奨 — 「毒ゼロコスト」の符号を戻す唯一の手。作成でなく主張(claim)に担保を課す。原資は評判のみ(N6でネイティブトークンは既に非目標化済み)。
3. 裁定は多数決でなく**外部権威ソース照合を主・再推論を従**とする(共通原因故障=§3.6 課題4を避ける全体の生命線)。

アンカー二層(時間軸、→ §4.2)とPoUW/ステーク(価値軸)は直交・非衝突。ステークは可変状態(mutable state)に載る。チャレンジ/裁定記録をアンカー連鎖ログに流せば、改竄不能な"裁判記録"を無償で得られる相乗効果もある。

### 4.4 直感④「明らか毒なら、ビットコイン式に奪う/消去する」→ 構造は正しい。全ては"明らか"の定義に集約

ビットコインが罰を機能させられるのは**不正が客観的に・安く検出できる**からである(二重支払いはチェーン上で可視)。オーナーの直感の構造そのものは正しく、問題は条件「これは明らか毒である」の**"明らか"を誰がSybilなしに決めるか**に集約される。

真偽を「投票するもの」と「客観的に反証できるもの」に分ける必要がある:

- 客観的に「明らか毒」と確定できる類(一次資料と矛盾/自己矛盾/トリプル破損/著作物丸写し=regurgitation/形式不正)→ 投票でなく外部の錨で毒判定でき、ビットコイン式に安全に「奪う・消去」が成立する。
- 客観的に確定できない類(安価な権威オラクルのない領域のもっともらしい誤り)→ "明らか"が安く成立せず、「奪う・消去」は兵器化する。

**兵器化の中身**: 「毒を捕まえたら奪える」という報奨は、正当なエントリを"毒だ"と偽って告発し、正直者の評判を奪ってエントリを消す動機(検閲失効に利益動機が付いた最悪版)を生む。裁定が主観/投票であれば「奪う」報酬が虚偽告発の燃料になる。LLM多数決裁定は共通原因故障(§3.6 課題4)を直撃し、多数派に反する正しい少数意見が毒として消され、人気の幻覚が生き残る(消去が真実を消し毒を守る)結果になる。

**同じ対称性**: 客観的に毒と証明できるエントリ ≈ 低価値エントリ。高価値で検証不能なエントリは"明らか"が成立しない領域。→ 「奪う・消去」はキャッシュが一番どうでもいい所で一番強く効き、一番大事な所で効かない。着地点は §4.3 の「救う最小構成」(ステーク+外部照合裁定、"明らか"を客観アンカーに限定)に合流する。

### 4.5 直感⑤(屋根)「毒はハルシネーション並みのノリでいい」→ 条件付きで採用可。会話で初の"解ける目標"

**再定式化の本質**: セキュリティ目標を「真実の保証」(達成不能)から「幻覚パリティ=生LLM以下に劣化させない非劣化目標」へ格下げする操作。

- **健全(詭弁ではない)**: 「絶対安全」でなく「基準線に対する非劣化」を目標に置くのはセキュリティ工学の標準作法(TLS的)。達成不能な目標を掲げ続けるより前進できる。**会話で初めて出た"解ける目標"。**
- **ただし理由付けの罠**: 「みんな幻覚を承知だから」を格下げの理由にすると詭弁になる。ユーザーが受容しているのは「自分が引いた確率的クジ」であって「他人が仕込んだ狙い撃ち」ではない。正しい理由付けは「真実の保証が不可能だから、達成可能な非劣化目標へ切り替える」である。
- **テールで破綻(最弱点)**: 定義を全体平均に取ると詭弁になる(安全な事実で分母を膨らませテールを薄める)ため、クラス別必須。測定は分布差(集中・持続・全員配信・不自然な一致)で敵対シグナル化でき可能ではあるが、**強制するとテールで割れる**: 低頻度・高被害の1件は統計が立たずベースラインも敵対シグナルも推定不能。しかも被害は非対称(同じ1件でも毒は全員へ署名付き高信頼で持続配信されるため被害期待値が桁違いになる)。→ **パリティのテールへの答えはない。テールは屋根の下に入れず別機構で守る。**
- **法的**: 「みんな浴びてる」という抗弁は署名付き配信網では弱まる(R4トレーサビリティと自己矛盾する)。故意注入はもはや幻覚ではなく意図ある虚偽情報の配信(名誉毀損・偽計業務妨害・薬機法等)であり、屋根は確率的幻覚のみに適用し、故意注入は屋根の外(ステーク+スラッシング+失効で対処、→ §4.3)とする。
- **屋根の効用/誤用**: 正しく使えば既存機構を「真実確立」→「敵対的寄与を自然ノイズの水面下に戻す」へサイズダウンできる(§3.6 課題4も「増幅されていないことだけ確認」に縮小して解け、独立検証も「不自然な一致の検出」に格下げできる)。誤用は全体平均で自己採点しテールを覆い隠す。**善悪はクラス別採点か平均採点かの一点で決まる。**

**最終判定**: 幻覚パリティは正当な統治的セキュリティ目標として採用**できる**。ただし条件付き — (a)評価をクラス別に、(b)故意注入を屋根の外に、(c)テールを別レイヤで守る。無条件の全体パリティは採用不可。

**強制可能な最小定式「2ティア + テール除外」**:

- **Tier-L(低リスク・パリティ適用)**: 一般常識・OSS仕様・歴史的事実・定義(permanent事実の大半)。目標=クラス別幻覚パリティ(このクラスの共有誤答率 ≦ 生LLMの同クラス幻覚率)。機構は軽量(共有ゲート+揮発性+独立検証を"敵対シグナル検出"にサイズダウン)。屋根が最も効く領域。
- **Tier-H(高リスク・パリティ禁止)**: 医療・法律・金融・安全(致死量・投資・与信・法的助言)、特定個人/法人を主語に持つトリプル(名誉毀損・GDPR)、**低頻度であること自体**をリスク要因に加える(統計が立たない=守れない)。目標=パリティ不適用。ステーク+外部権威照合裁定+失効をフル適用、またはそもそもPublic共有しない(Private/都度推論)。共有ゲートの「疑わしきは共有しない」を最大限保守側へ寄せる。
- 要点: パリティを"統計が立つTier-L"に閉じ込めて強制し、Tier-Hはパリティ管轄外と宣言して別機構へ渡す。「平均で薄める」余地が構造的に消える。

### 4.6 「PoCを1から再設計すべきか」→ 全面書き直しは不要。様変わりしたのはコードでなく戦略

**結論: 全面書き直しは不要。** オーナーの「様変わりした」体感は本物だが、変わったのは「何を守るか(真実→幻覚パリティ)」「どう守るか(アンカー/ステーク/ティア)」という**要件と統治方針**であって、現PoCが実装する**コアループの骨格は一度も否定されていない**。

**現物の仕分け**(`poc/src/` 全ソースを読んだ上で):

- **そのまま生き残る(資産)**: `main.rs` の配線思想(検索→ミス→推論→判定→署名→登録→verify)、**trait 3本(Embedder/Signer/Agent)+factory方式**、`triples.rs` の型設計(`PredicateEntry`/`VolatilityClass`/`TripleDecomposition`)、`volatility.rs` の非対称原則。trait 3本を捨てるのは自傷行為。
- **手を入れる(構造は残し中身を作り直す)**: `cache.rs` の `CacheEntry`/`signed_payload()`(**再設計の主戦場 = §2.3 改良案A・B・D**)、`verify()`/`load()`(shareable/embeddingを信頼せず受信側再導出、→ §3.6 課題2)、`pipeline.rs` の `judge_entry`(Tier分類の追加、→ §4.5)。総当たり検索は急がない(PoC規模ではO(n)が正しい)。
- **捨てて作り直す**: witness/評判/ステーク/失効 — だが大半はそもそもまだ書かれていない。「作り直し」の実体は既存コードの破棄ではなく未着手部分の新規設計。既存コードで実際に作り直すのは `cache.rs` のエントリ形式1点に集中する。

**「1から」が正当化される3条件**: (1)trait境界が誤り→該当せず(trait 3本は正しい)、(2)言語/プラットフォーム変更→部分該当(`poc/` → `src/core/` だが"昇格"であって書き直しではない)、(3)データモデル根本非互換→ `cache.rs` のエントリ形式のみ該当。オーナーの「様変わり」体感は(2)(3)が同時に視界に入ったことが原因で、骨格(1)は無傷。

**「動く的撃ち」vs「第2PoC」**: 「戦略未決で書き直すのは動く的撃ち」という懸念は現物で裏付けられる(エントリ形式はPublic/Company判断の下流。Company単独ならwitness/ステークがほぼ不要でエントリ形式が大幅に簡素化する)。「第2PoCを別に立てる方が速い」は部分的にのみ成立する(現PoCは単一ノードでS3検証に構造的に使えず、改ざんデモも自鍵の自作自演で多ノード脅威を再現しない=正しい/だがtrait 3本ごと捨てるなら誤り)。調停案は白紙にせず、既存資産を移植して `src/core/` に第2の器を作ること。

**推奨シーケンス**:

- **[A] 今すぐ決める(コードは書かない・戦略決定)★最優先** — ①Public層かCompany層か、②幻覚パリティ(2ティア、→ §4.5)の採用可否、③アンカー/ステーク(→ §4.2・§4.3)をS3/S4スコープに入れるか。決定をdocsに落とす。
- **[B] 戦略確定後、的を絞って作り直す(S2.5)** — `cache.rs` のエントリ形式のみ(§2.3 改良案A・B・D・E)。まだ `poc/` 上・単一ノードでよい。triples/volatility/pipelineはほぼそのまま接続する。
- **[C] S3の器として `src/core/`(Rust)を新規に立てる = 「第2PoC」の正体** — `poc/` からtrait 3本+triples/volatility/pipeline+新エントリ形式を移植。**`poc/` はS1/S2の参照実装として凍結・保存する(消さない=リグレッション基準)**。`src/core/` でwitness/アンカー/多ノードを新規実装。CLAUDE.md既定(core=Rust, UI=Blazor)に沿う。
- **[D] 保留** — 評判/ステーク/裁定(S4)・失効/regurgitation(S5)・UI(S6)は戦略確定まで触らない。
- **親のシーケンスへの補正**: S2.5(エントリ形式)の直前に「戦略ゲート[A]」を必ず挟む(エントリ形式はPublic/Company判断の下流だから)。

**今はコードを書く時期か → 書く時期ではない。docs(戦略決定)に留まるべき局面**: ①最も安いコストで最大の不確実性を潰せるのが今の戦略決定(Public/Company・パリティ採否はコード0行で下流全体を規定する)、②現PoC(S1/S2)は「コアループは成立するか」に既にYesと答えた(71 tests passed)、③今書けば書くほど手戻りが増える。ただし「留まる=止まる」ではない — 次の具体的アクションは、**この対話ログの結論をArchitecture.mdとRoadmap.mdに正式反映すること**(現状は会話ログにしか存在せず、docsが源泉真実というCLAUDE.md原則から乖離している)。本節への集約と、[Roadmap.md](./Roadmap.md) §5 の索引更新が、その第一歩にあたる。

### 4.7 主戦場の決定:Company先行 → Public後続の段階展開【★オーナー採用決定 2026-07-17】

**この項は §4.1〜§4.6 と異なり、レビュー指摘・検討記録ではなく、オーナーが戦略として採用決定した事項(戦略ゲート[A]の決着)である。** 本体docs([Architecture.md](./Architecture.md) §9 / [Roadmap.md](./Roadmap.md))への反映対象であり、実際に反映済み(出典として本節を参照)。ただし段階展開の内側の個別機構(アンカー二層/ステーク+外部照合裁定/witness具体設計/Tier-H裁定の具体設計)は依然「未採用の候補」であり、Public Phase2で改めて判断する。

**Public vs Company の8軸比較 → 全軸でCompany優位(僅差でない)**

価値提案/ヒット率/脅威・シビル・毒/著作権・プライバシー・法務/インセンティブ/既存資産/製品化までの時間/上流陳腐化 — すべてCompany優位。理由: Public層の困難のほぼ全て(シビル・毒・法務・インセンティブ・ヒット率)はpermissionlessと匿名多数から生じ、Companyはその両方を消す。

**採用した戦略 = Company先行 → Public後続の段階展開。** fable判定は「逃げでなく正しい漸進」である: Public層の困難はCompanyで"解けない"のでなく"存在しない"(存在しない問題を先送りするのは逃げでなく問題の切り分け)。Companyで作る背骨はPublicでもそのまま使う土台であり、実データでゲート通過率・ヒット率・幻覚パリティ(→ §4.5)を実測できる=Public設計への偵察になる。

**逃げに転落する2経路(踏んだら逃げ)**

1. 「Publicをやると言い続けながらやらない」— ただし「Companyで十分と判断して止める」のは逃げでなく良い経営判断(Phase1完了時に実データでPublic進出の是非を再評価でき、進まない自由が確保される)。
2. Company都合でエントリ形式を鋳込み空スロットを空け忘れる(→ Public移行が作り直しに化ける)。

**二律背反の捌き方 = 「省略」を「削除」でなく「未実装スロット」に**

Companyでは実装しないが、Publicで必要になるものの"場所"だけエントリ形式に確保する。`witness_sigs`/`anchor_proof`/`stake`/`trust` はCompany Phase1では空(optional)だが、署名対象の内外の区別と可変状態側への配置(→ §2.3 改良案B)だけ今決める。→ **改良案A・B・D(→ §2.3)は「後で足す機能のための余白を型で保証」するためにこそ必須**(段階展開を採るなら"必須前提"に格上げ)。

**共有コア(背骨) vs fork必須の境界**

- 背骨(両フェーズ共通): trait 3本 / エントリ形式の不変コア / triples / volatility / 共有ゲート(Tierフィールドを空で用意) / 検索(trait境界同一)。
- fork必須(だが場所は確保): witness+アンカー二層(→ §4.2) / 評判+ステーク+裁定(→ §4.3・§4.4) / revocation+regurgitation。

**戦略ゲート[A]残り2項目の仕分け**

- 幻覚パリティ(→ §4.5): 概念(Tier分類)はPhase1から / 強制機構(Tier-H裁定)はPhase2から。
- アンカー/ステーク(→ §4.2・§4.3): Phase2に全繰り延べ。ただしエントリ形式に `anchor_proof` / `stake` の可変フィールドを空で確保。
- → **S2.5エントリ形式の確定条件**: 不変コアはPublicまで見据えて今フル確定してよい(question正規化/facts/provenance〔agent署名対象化〕/created〔UTC〕/初期volatility/Tierタグ)。可変状態は空スロットを型で宣言。

**更新シーケンス([A]決着で[B]がアンロック)**

- **[A] 戦略ゲート** = 決着(Company先行段階展開 / パリティ概念Phase1・強制Phase2 / アンカー・ステークPhase2繰り延べ+スロット確保)。残 = docs反映(→ 本節・[信頼性設計メモ.md](./信頼性設計メモ.md) §9・[Architecture.md](./Architecture.md) §9・[Roadmap.md](./Roadmap.md))。
- **[B] S2.5 エントリ形式の確定** = 今すぐ着手可能(不変コアをPublicまで見据えフル確定+可変状態の空スロット宣言。`poc/` 上で単一ノードのまま)。
- **[C] `src/core`(Rust)へ昇格** = Company Phase1本体(背骨を移植、S3相当を"社内版"に縮約=多ノード共有はやるがwitness/アンカーは共通時計で代替、S6モード分離、`src/ui`着手可、`poc/` は凍結・参照実装)。
- **[移行ゲート]** Phase2条件: 不変コア安定/ゲート通過率の社内実測/Tier-L非劣化実測/空スロット設計が机上で閉じる/弁護士レビュー着手。
- **[D] Public Phase2** = 空スロットを埋める(作り直しでなく追加): witness+アンカー / 評判ステーク+外部照合裁定 / Tier-H強制 / revocation+regurgitation / PoW ID / 段階公開(S7)。

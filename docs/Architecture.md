## 1. 概要

同種・類似の質問に対しLLMが毎回推論する無駄を、**「まずキャッシュを意味検索し、ミス時のみ推論する」**構成で削減する。キャッシュは分散型P2Pネットワーク上で人類規模に共有する。共有対象は **「文脈自立 × 事実型(permanent/slow)」** の質問回答に限定し、中央管理者を置かない前提でシビル攻撃・キャッシュ毒・鮮度劣化・詐称・法的リスクに構造的に耐える。

**一言**: AIに毎回考えさせず、人類が過去に考えた結果をまず探し、無い時だけ考えさせる。その蓄積を分散ネットワークで共有する。

**[補完]** 展開戦略: 上記の「人類規模の共有」はPublic層の最終像であり、実装はCompany層(社内限定)をPhase1として先行させ、Public層をPhase2として後続させる段階展開を採る(→§9 [補完]、一次記載は[信頼性設計メモ.md](./信頼性設計メモ.md) §9)。

**[補完]** プロダクトの性格づけ(2026-07-19): 「雑学キャッシュ」に留めるか「共有知識キャッシュ」を狙うかは二択ではなく**段階**である。雑学(宣言的事実)は失敗が良性で信頼機構の訓練場として最適だが、推論コスト低下で経済価値が侵食されるためゴールではなく、本設計の本当の資産は雑学の中身でなく**署名・再導出・揮発性・信頼度という検証インフラ**である。正確な性格づけは「**共有知識キャッシュを目指すが、信頼機構(S4以降)の成熟度がカバー範囲を規定する**」。一次記載は[信頼性設計メモ.md](./信頼性設計メモ.md) §9「プロダクトの性格づけ」追記(雑談=原理的に不適格/コード=適格だが時期尚早、の区別を含む)。

## 2. 設計目標 / 非目標

### 2.1 設計目標

| # | 目標 | 根拠 |
|---|---|---|
| G1 | GPU/API推論回数の削減(電力・レイテンシ・コスト) | 元コンセプトの主目的 |
| G2 | ユーザー横断で共有される意味キャッシュ | 競合(GPTCache/MeanCache)に対する唯一の空白地帯・本丸 |
| G3 | 中央管理者なしでの信頼性担保 | 分散型ネットワークを名乗る以上の必須要件 |
| G4 | ネットワーク物理分離によるプライバシー | AI判定でなく参加ネット自体を分ける |
| G5 | 法的距離の構造的確保(Winnyとの差別化) | 「事実トリプルのみ・失効可能・出所記録」をプロトコルに焼き込む |

### 2.2 非目標(やらないこと)

| # | 非目標 | 理由 |
|---|---|---|
| N1 | P2Pネットワーク自体が推論する | 推論はAgent層へ委譲。ネットワークは保存・検索・検証のみ |
| N2 | 時事・文脈依存・主観・個人情報の共有 | 脅威B(文脈欠落)・D(鮮度)を構造的に回避するため対象外 |
| N3 | 全順序合意(ブロックチェーン型コンセンサス) | 結果整合+複数バージョン併存で十分。合意コストは過剰 |
| N4 | シビル攻撃のコストゼロ完全排除 | 理論上不可能(Douceur 2002)。目標は「割に合わなくする」 |
| N5 | 長文・逐語表現の保存 | 法的リスク源。保存は事実トリプル+出典メタのみ、回答は受信側で再合成 |
| N6 | ネイティブトークン/ステーク経済圏 | 投機・規制・匿名参加の思想と衝突(§9 PoS参照) |
| N7 | 匿名性を売りにすること | Winny判例で最も不利に働いた要素。売りは「説明可能性」 |

**[補完]** 上表のPublic層関連の非目標(特にN2/N6/N7)は、Company層(Phase1)・Public層(Phase2)いずれにも共通して適用される制約である。ただし主戦場はCompany層を先行させる段階展開を採用しており(§9参照)、Public層自体の着手はPhase2条件付きとなる。詳細は[信頼性設計メモ.md](./信頼性設計メモ.md) §9・[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.7を参照。

## 3. 用語定義

| 用語 | 定義 |
|---|---|
| エントリ (Entry) | キャッシュの最小単位。質問群に対する1回答候補+事実トリプル+署名+witness+揮発性タグ+出所メタ |
| 意味検索 | 質問をEmbedding化し、コサイン類似度で近い質問群を同一視する検索 |
| 文脈自立 | 前提の会話なしで単独回答できる質問の性質(軸1) |
| 事実型 | 回答が事実トリプルへ分解可能で、主観・時事でない性質(軸2) |
| 揮発性タグ (volatility) | permanent / slow / volatile の鮮度クラス。確信度付きの確率的推定として保持 |
| witness署名 | 受信ノードが「最初に受信した時刻」を署名付き記録したもの。分散タイムスタンプ |
| author署名 | 投稿ノードが秘密鍵で付す「誰が言ったか」の固定 |
| 事実トリプル | (主語, 述語, 目的語) の知識断片。例: (Winny, 開発者, 金子勇) |
| 独立検証 | 異なるAgent/時期/ノードが独立生成した回答のトリプル一致率で信頼度を測ること |
| スラッシング | 誤情報が露見した登録ノードの評判を焼失させる罰 |
| 局所EigenTrust | グローバル合意せず各ノード近傍でのみ信頼を割り引き伝播する評判計算 |
| revocation | 署名付き失効通知。フラッディングで各ノードが自律配信停止 |
| regurgitationフィルタ | 既存著作物の逐語再現を検知し保存を拒否するフィルタ |
| question_key **[補完]** | 検索・重複排除・版束ね用の質問キー。`sha256("nyllm/qkey/v1\n" + fold(質問))`。entry_id(内容アドレス)とは二層分離(S2.5、→§6 [補完]) |
| immutable_core / mutable_state **[補完]** | エントリの二分構造(S2.5)。前者=署名・entry_id対象で作成後不変、後者=署名対象外で各ノードがロード時に再導出・更新する(送信者の値は信頼しない) |
| Tier-L / Tier-H **[補完]** | 幻覚パリティ2ティア。Tier-L=低リスク(パリティ適用領域)/ Tier-H=医療・個人情報等の高リスク(パリティ禁止・別機構へ)。一次記載は[信頼性設計メモ.md](./信頼性設計メモ.md) §9 |
| initial_tier / tier_operative **[補完]** | 著者の初期ティア主張(core署名対象=事後説明責任の固定)/ 受信側がロード時に再導出する運用ティア(→§6 [補完]・§7 [補完]) |
| anchor_proof **[補完]** | 外部時刻アンカー参照。witness循環を断つアンカー二層構想に対応するPhase2空スロット(mutable_state側) |
| stake **[補完]** | 評判ベースのステーク(Phase2空スロット)。ネイティブトークンではなく、非目標N6と矛盾しない |

## 4. 全体アーキテクチャとデータフロー

### 4.1 レイヤ構成

```text
┌──────────────────────────────────────────────┐
│  Client / UI  (モード別アイコン: Public/Company/Private) │
└───────────────┬──────────────────────────────┘
                │ 質問
                ▼
┌──────────────────────────────────────────────┐
│  NPU Layer                                    │
│   Embedding生成 / 意味検索 / 類似度判定 / 分類  │
└───────────────┬──────────────────────────────┘
                │ 質問ベクトル
                ▼
┌──────────────────────────────────────────────┐
│  Semantic Cache Layer (P2P)                   │
│   分散ストア / DHT / 署名・witness検証 / 失効   │
└──────┬───────────────────────────────┬───────┘
       │ Hit                           │ Miss
       ▼                               ▼
  回答再合成して返却            ┌─────────────────┐
  (トリプルから)                │  Agent Layer     │
       ▲                        │ OpenAI/Claude/   │
       │                        │ Gemini/Local/社内 │
       │                        └────────┬────────┘
       │                                 │ 回答
       │                                 ▼
       │                        ┌─────────────────┐
       │                        │ 判定パイプライン  │
       │                        │ L0/L2ゲート(§7)  │
       │                        └────────┬────────┘
       │                                 │ 昇格可
       │                                 ▼
       │                        署名+witness+トリプル化
       │                        して Cache へ登録
       │                                 │
   ┌───┴─────────────────────────────────┴───┐
   │  Reputation Layer (横断)                  │
   │  3層評判 / 独立検証 / スラッシング         │
   └──────────────────────────────────────────┘
```

### 4.2 読み取り hot path(毎クエリ・軽量)

```text
質問
 → Embedding生成(NPU)
 → 共有キャッシュ意味検索(τ≥0.9)
 → Hit:
      候補エントリ群を取得
      → 署名/witness/失効チェック
      → 揮発性タグ判定(permanent=日付無視 / slow=作成日+猶予)
      → 層1/層2で信頼度スコアリング(既存の集計値を参照するだけ)
      → 確率pで層3抜き打ち再推論(§8.2)
      → トリプルから回答を再合成して返却
 → Miss: 書き込み cold path へ
```
原則: hot pathでは重い判定を行わない。既に計算済みの集計値を引くだけ。判定は書き込み時に1回。

### 4.3 書き込み cold path(ミス時のみ・重量)

```text
Miss
 → Agent層へ推論委譲(複数Agent可)
 → 回答生成
 → 判定パイプライン(§7)
      L0 ルール足切り(文脈依存語/揮発語/主観語/個人参照)
      L2 Agent自己申告 + 案4トリプル分解
 → 全ゲート通過 かつ volatility∈{permanent,slow} のみ昇格
 → regurgitationフィルタ(§11)
 → 事実トリプル抽出 + 出典メタ付与
 → author署名 → ネットワーク配布
 → 受信ノードがwitness署名(受信時刻保証)
 → 独立検証キューへ(既存トリプルとの一致率更新)
```

## 5. コンポーネント仕様

### 5.1 NPU層

| 項目 | 仕様 | 根拠/[補完] |
|---|---|---|
| 役割 | Embedding生成・意味検索・類似度判定・登録時分類。**推論はしない** | 元コンセプト |
| Embeddingモデル | MPNet級(768次元)を推奨 | MeanCacheの実証。Llama-2は不適(F1=0.75/30GB)で回避 |
| 圧縮 | PCA 768→64次元(約83%削減) | MeanCache採用手法 |
| 共有用しきい値 | τ ≥ 0.9(ローカルより高精度側) | 誤ヒット(脅威A)回避。偽陽性が人類規模で汚染するため精度優先 |
| ローカル用しきい値 | τ ≈ 0.8 前後 | MeanCache(MPNet 0.83) |
| context-chain照合 | メタに文脈鎖を保持し照合 | MeanCacheから借用。文脈自立判定の補助 |

### 5.2 Semantic Cache層 (P2P)

| 項目 | 仕様 |
|---|---|
| トポロジ | 構造化P2P(DHTベース)を推奨。ノードはエントリを保持・中継・検証 |
| キー | `hash(正規化質問群 + Agent + Model)`(案2)。改ざん**検知**用であり詐称防止ではない(§8)。 **[補完]** 実装済み定義はS2.5で `entry_id = sha256(core_bytes)` + 検索・重複排除用 `question_key` の二層に置換済み(→§6 [補完] 差異表) |
| 保存単位 | §6 のエントリスキーマ。**長文でなく事実トリプル+出典メタのみ**を保存 |
| バージョニング | 同一質問に複数回答版を併存(案1)。タグも版ごとに保持 |
| 検索 | Embedding近傍検索 → 候補群 → 署名/失効/信頼度でランク |
| 失効 | revocationフラッディングで各ノードが自律的に配信停止(§11) |

### 5.3 Agent層

| 項目 | 仕様 |
|---|---|
| 役割 | ミス時の推論委譲先。ネットワークは推論しない(N1) |
| 対象 | OpenAI / Claude / Gemini / Llama / ローカルLLM / 社内LLM |
| Public層のモデル選定 | **再配布を明示許可するライセンスのモデルに限定**。商用API出力はCompany/Privateに留める(§11 規約マトリクス) |
| 独立性 | 独立検証のため、可能なら異なるAgent/時期で複数生成し一致率を測る |
| 自己申告 | 登録時に「単独回答可能か」「事実型か」「volatility」を申告。ただし**決定でなく一票**(§7) |
| 実装状況(選択可能な推論先) **[補完]** | `OllamaAgent` 実装済み(`feature = "ollama"`、`src/core/agent.rs` + `src/core/agent/ollama_agent.rs`、`ureq`経由、既定モデル `gemma3`)。バックエンドは `AgentConfig`(環境変数 `NYLLM_AGENT_BACKEND=mock\|ollama` 等)で選択。`Agent::ask` は `Result<String, AgentError>` 化され、Agent失敗時はjudge/登録/announceを行わない。一次記載: [実装スペック(2026-07-18、改訂注記あり)](./superpowers/specs/2026-07-18-selectable-inference-backend-design.md) |
| provenance追跡 **[補完]** | `OllamaAgent::name()` は `ollama:<モデル名>` を返し、provenanceからモデル単位まで追跡可能(§11 R4「出所の完全記録」と整合) |

### 5.4 評判層 (Reputation)

§8 に詳述。3層(エントリ内在信頼度 / ノード評判 / 消費側最終防衛)。全レイヤを横断し、hot pathには集計値を、cold pathにはスラッシング/貢献記録を提供する。 **[補完]** 本層の実装の大半はPublic Phase2繰り延べ(→§8冒頭[補完]・§9 [補完])。うち**層1(エントリ内在信頼度=`independent_agreement`/`supporting_versions`)のみCompany Phase1で先行実装済み(2026-07-19)**: 作用は検索ランキングのタイブレーク(既定重み0の実測ゲート)+API/UI表示の**助言のみ**であり、共有ゲート(`shareable`)には配線しない。層2(局所EigenTrust等)・層3(抜き打ち再推論)はPhase2据え置きのまま。一次情報は [Roadmap.md](./Roadmap.md) §2 S4節・[S4_Company_Phase1_層1内在信頼度先行設計.md](./S4_Company_Phase1_層1内在信頼度先行設計.md)(冒頭「改訂注記(2026-07-19)」)。

### 5.5 失効 (revocation) 機構

**[補完]** 本機構の実装はPublic Phase2繰り延べ(→§9 [補完]、[Roadmap.md](./Roadmap.md) §0対応表 S5)。Phase1ではドメイン分離タグ `nyllm/revocation/v1` の予約のみ([S2.5_エントリ形式設計.md](./S2.5_エントリ形式設計.md) §11)。機構の設計自体は以下のとおり有効なまま保持する。伝播・権限モデル・S3残課題(ソース側フィルタ非対称)への解消案を含む先行設計は [S5_Public_Phase2_法的機構先行設計.md](./S5_Public_Phase2_法的機構先行設計.md) §3を参照(先行設計・実装未着手。段階定義・ゲートは不変)。

| 項目 | 仕様 |
|---|---|
| トリガ | 著作権侵害通知 / 誤情報露見 / タグ僭称露見 |
| 形式 | 署名付き失効通知(対象エントリhash + 理由コード + 発行者署名) |
| 伝播 | 中央なしでフラッディング。各ノードが受信後に自律配信停止 |
| 評判連動 | 失効されたエントリの登録ノードをスラッシング |
| 免責要件 | 「通知を受けたら止められる仕組み」は日米EU共通の免責入口(§11) |

## 6. キャッシュエントリ データモデル

```jsonc
{
  // --- 識別 ---
  "entry_id": "a3f9c2...",            // hash(正規化質問群 + Agent + Model) 改ざん"検知"用
  "network": "public",               // public | company | private
  "schema_ver": 1,

  // --- 質問側 ---
  "question_cluster": [              // 意味的に同一視する質問群
    "Winnyとは？", "Winnyって何？", "P2PソフトWinnyについて教えて"
  ],
  "embedding_pca64": [/* float32 x64 */],  // PCA圧縮済みベクトル

  // --- 回答側: 長文でなくトリプル+メタのみ保存(N5, §11) ---
  "facts": [
    {"s": "Winny", "p": "開発者", "o": "金子勇"},
    {"s": "Winny", "p": "種別",   "o": "P2Pファイル共有ソフト"}
  ],
  "answer_template_ref": null,        // 回答は受信側で再合成。逐語文は保存しない

  // --- 鮮度: 確率的推定として保持(§10) ---
  "volatility": {
```

## 7. 共有対象の判定パイプライン(L0/L2ゲート)

### 7.1 判定式

```text
共有可能 = 文脈自立(軸1) AND (事実型 かつ volatility∈{permanent, slow})(軸2)
デフォルト = 共有しない。全ゲート通過時のみ Public/Company へ昇格。
```
根拠: 人類規模の汚染は偽陽性が偽陰性より遥かに有害 → 精度優先、迷ったら共有しない。

### 7.2 判定タイミング

| 経路 | タイミング | 処理 | コスト |
|---|---|---|---|
| 読み取り(hot) | 毎クエリ | 引くだけ。重い判定なし | 極小 |
| 書き込み(登録) | ミス+回答後 | フル判定して共有フラグ決定 | 許容(ミス時のみ) |

### 7.3 ゲート詳細

**軸1: 文脈自立(安い順)**

| 層 | 手法 | 除外例 |
|---|---|---|
| L0 語彙 | 指示語・代名詞の裸出現 | それ/その/これ/さっきの/上記の, it/that/above |
| L0 省略命令 | 対象省略の命令形 | 赤に変えて / もっと短く / 続けて / 次は？ |
| L2 Agent自己申告 | 「前提会話なしで単独回答できるか?」 | No → 除外 |

**軸2: 事実型/揮発性**

| 層 | 手法 | 除外/分類 |
|---|---|---|
| L0 揮発語彙 | 時事トリガー | 最新/現在/今/今日/価格/株価/天気/2026年 → volatile |
| L0 主観語彙 | 意見・主観 | おすすめ/べき/どう思う/best/should → 除外 |
| L0 個人参照 | 一人称・自分データ | 私の/自分の/このファイル → Private |
| L2 Agent+案4 | 回答を事実トリプルへ分解 | 分解可 → factual。生成文/分解不能 → 除外 |

### 7.4 登録時フロー

```text
回答生成完了
  → [L0] 文脈依存語/揮発語/主観語/個人参照 を含む? ── 含む → 共有しない
  → [L2] 単独回答可? 事実型? volatility=? ── No or volatile → 共有しない
  → 全Yes かつ permanent|slow
  → regurgitationフィルタ通過?（§11）── 不通過 → 共有しない
  → 事実トリプル抽出 → 署名+witness付きで昇格登録
```
L0で大半を高精度に足切りし、生き残りだけL2で精査。NPU/GPUをほぼ使わず省電力思想と一致。

**[補完] S2実装での保守的厳格化(2026-07-17)**: 実装(S2)では上記ANDゲートに「案4分解の全文成功」「全述語が述語オントロジー収録済み(allowlist)」の条件を追加し、さらにコピュラ定義述語(種別/is-a)でも目的語に時事シグナル(数値・通貨・時点語・年)を含む場合はpermanent昇格を抑止してvolatileへ倒す安全策を加えている。いずれも判定を緩めない方向のみの変更。通過率実測(最終共有可 4/12 等)の一次情報は [Roadmap.md](./Roadmap.md) S2節を参照。

**[補完] ティアの位置づけ(幻覚パリティ2ティア)**: 登録時に著者は、本節の判定結果とあわせて `initial_tier`(Tier-L/Tier-H)を初期主張としてエントリの immutable_core に固定する(署名対象=事後説明責任の基礎)。受信側はこの値を運用判断に信頼せず、ロード時に `tier_operative` を再導出する([S2.5_エントリ形式設計.md](./S2.5_エントリ形式設計.md) §6手順8。§6 [補完] 差異表と整合)。ティア分類器は未実装で既定はTier-L(共有ゲートへ配線する段階で「未分類はTier-H(保守側)」へ反転予定)。強制機構(Tier-H裁定)はPublic Phase2(→§9 [補完])。統治目標の一次記載は[信頼性設計メモ.md](./信頼性設計メモ.md) §9「統治目標の再定義」。

入れ子関係は 共有適格 ⊂ 永続ローカル適格 であり、その外側の発話はセッションスコープに閉じる。キャッシュ可能性の本質条件は「**回答が質問の安定な関数であること**」で、主観/客観(§7.3軸2の主観語彙)はその不完全な代理変数にすぎない — 切るべき軸は主観性ではなく**発話行為タイプ**(情報要求 vs 挨拶・対話行為・生成依頼。例:「調子どう?」はL0語彙表のどれにも当たらないが情報要求ではないためキャッシュ不適格)。Phase1の現実解としてはセッション層を新設せず、「雑談・文脈依存発話はそもそもキャッシュ登録しない(記録もしない)」**二層構成**でも越境事故は防げる(セッション層は必要になってから足す)。なお S2.5 形式は answer 平文を保存しない(core=factsのみ)ため、facts が空になる雑談で「古い定型文がヒットして返る」事故は現構造では起きない — ただし将来ローカルに answer 平文保存を再導入する場合、それは S2.5 が著作権面(§11 R1/N5)・毒面積縮小のために捨てたものの部分復活であると自覚的に扱うこと。

**[補完] 既知の穴: lookup にTTL・クラス別制御が皆無だった問題(2026-07-19 記録)**: 設計は「volatile はローカル短期TTLのみ」と規定し([信頼性設計メモ.md](./信頼性設計メモ.md) §5、本書§9表のPrivate行、実装 `share_gate` の判定理由文も同文言)、§8.5 も slow=作成日+猶予 の鮮度判定を規定するが、これらを lookup 時に強制する機構は段階的にしか存在していない。設計原則は「**登録時ゲートは情報を不可逆に捨てる。TTL・クラス別しきい値は lookup 時に強制すべき(後からチューニング可能で、判断の変更が既存エントリへ遡って効く)**」(一次記載は[信頼性設計メモ.md](./信頼性設計メモ.md) §5追記)。この検査は軽量フィルタであり §7.2「読み取り(hot)は重い判定なし」の原則とは矛盾しない。

## 8. 信頼性メカニズム(実装視点)

**[補完] Phase帰属**: 本節の機構のうち、witness署名・層2ノード評判(局所EigenTrust・ID生成PoW)・層3抜き打ち・スラッシング・投票重み付けの実装はPublic Phase2に繰り延べる(Company Phase1では層1=トリプル一致率相当のみ先行しうる。→§9 [補完]、[Roadmap.md](./Roadmap.md) §0対応表 S4)。機構の設計自体は有効なまま保持する。

**[補完] 層1の実装状況(2026-07-19)**: 層1はCompany Phase1で`src/core`へ先行実装完了した。一致率は正規化トリプル集合の**版ペア間Jaccard平均**(同一`question_key`の版集合を独立生成回答群とみなす)。§8.2の「witness署名時系列による独立性検証」はPhase2据え置きのため未実装であり、その代わりの保守的規約として、facts分解が成功している版が0または1(一致率のペアが組めない)場合は `independent_agreement = 0.0` とし「独立の裏づけ未取得」を全版一致(1.0)と区別する。`trust`は各ノードがローカルで再導出し、転送(Transfer/エンベロープ)では運ばない(送信者値不信任)。作用は助言のみ(検索ランキング既定重み0+API/UI表示)で、共有ゲートには配線しない。算出ロジックはポリシー差し替え点(policy hook)の背後にあり、Phase2はwitness独立性検証付きの算出へ差し替える。詳細は [Roadmap.md](./Roadmap.md) §2 S4節・[S4_Company_Phase1_層1内在信頼度先行設計.md](./S4_Company_Phase1_層1内在信頼度先行設計.md)(冒頭「改訂注記(2026-07-19)」)。

### 8.1 全体思想

**投票を「真偽の決定」から「探索の優先度ヒント」へ格下げする。** シビル攻撃は多数決を攻撃するもの → 真偽を多数決で決めない。事実型限定という線引き(§7)がそのまま土台になる。事実は独立検証で決められるため。

**[補完] 統治目標の再定義(2026-07-17)**: 本節の到達目標は「真実の保証」ではなく「**幻覚パリティ(スタンドアロンLLMに対する非劣化)+テール除外**」である。主要LLMは学習コーパスを共有し同じ誤りに独立到達するため、独立検証のトリプル一致率が測るのは「真実」ではなく「学習データの合意」であり、真実の保証は達成不能。したがって本節の独立検証・評判機構は「真実の確立」ではなく「敵対的寄与を自然ノイズの水面下に戻す(不自然な一致・増幅の検出)」機構として運用する。評価はクラス別(Tier-L)に限り、Tier-H(医療・個人情報等の高リスク・低頻度領域)はパリティ管轄外として別機構(Phase2: ステーク+外部照合裁定+失効)で守る。一次記載は[信頼性設計メモ.md](./信頼性設計メモ.md) §9「統治目標の再定義」、採用経緯は[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.5。

### 8.2 3層評判

```text
層1 エントリ内在の信頼度(シビル非依存)★主軸
    独立生成された複数回答のトリプル一致率(entry.trust.independent_agreement)
    + witness署名時系列で"独立性"を検証(自作自演の同時注入を検知)
層2 ノード評判(局所EigenTrust)
    誕生証明(witness署名を流用) + ID生成PoW
    検証済み貢献を対数重みで累積 / 誤情報でスラッシング
    グローバル合意せず各ノード近傍でのみ信頼を伝播
層3 消費側の最終防衛
    高信頼キャッシュでも確率pで抜き打ち再推論 → 層1/層2へフィードバック
```

### 8.3 シビルを割に合わなくする(コストの掛け算)

| コスト | 効き方 | 実装 |
|---|---|---|
| ID生成PoW | 限界コストを非ゼロに(補助のみ) | 公開鍵ハッシュに難易度付きHashcash(数分) |
| 時間(最重要) | 新規IDは重みゼロ開始。半年分の貢献履歴は並列化不可 | 誕生証明時刻(witness署名)からの経過を重みに反映 |
| 貢献 | 重み獲得の唯一の道=「後に正しいと検証される回答」 | 独立検証で正と確認された登録数を対数で加算 |
| スラッシング | 稼いだ評判は毒注入1回で焼失 | 失効/僭称露見でreputationを大幅減 |

### 8.4 投票の重み付け(二次防衛)

- 票の重み = `f(ID年齢, 検証済み貢献, ローカル信頼)`、対数(上に凸)で単一大物ノードの支配も抑制
- 否定票(反証つき=矛盾する検証済み事実へのポインタ)は肯定票より強く効かせる
- 票自体を希少化(投票へのPoWスタンプ / 評判比例の投票上限)

### 8.5 鮮度と詐称防止(統合)

- **詐称防止**: ハッシュは改ざん**検知**のみ。詐称防止は author署名(誰が)+ witness署名(いつ受信を第三者が保証)の2点で担保。未来日詐称はwitness時刻で棄却。
- **鮮度**: `claimed_date`(**[補完]** S2.5実装では core の `created` に対応 →§6 [補完] 差異表)+ `volatility.class`。permanent=日付無視、slow=作成日+猶予、volatile=共有非対象。witness時刻列で「いつから同じ回答が安定しているか」を観測しタグを裏付ける。

### 8.6 既存技術の適否(採否の確定)

| 技術 | 採否 | 理由 |
|---|---|---|
| EigenTrust | 部分採用 | 「信頼を評価者の信頼で割り引き伝播」の核心のみ。pre-trusted peersは誕生証明+貢献履歴で置換。収束計算は局所化 |
| PoW(ID生成時) | 補助採用 | 限界コスト付与のみ。継続的PoWは電力浪費で不採用 |
| Proof-of-Stake | 不採用 | ステーク資産なし。トークンは投機/規制/匿名参加思想と衝突 |
| Web-of-Trust | 補助採用 | ローカル信頼の初期ブートストラップ(紹介)に限定。手動署名依存で主軸不可 |
| SybilGuard/Limit/Rank | 条件付き | 「長期取引履歴」を擬似社会グラフに流用可 |
| ブロックチェーン全順序合意 | 不採用 | 結果整合+複数版併存で十分。合意コストのみ輸入することになる |

## 9. ネットワークモード分離設計

プライバシーはAI判定でなく**参加ネットワークの物理分離**で担保する。

| モード | 範囲 | 用途 | 共有ポリシー | 起動 |
|---|---|---|---|---|
| Public | 全体共有 | 一般知識/OSS/公開情報 | §7全ゲート通過+再配布可ライセンスのモデルのみ | `ai-node --mode public` |
| Company | 社内限定 | 社内FAQ/ナレッジ/RAG | 社内配布のみ。商用API出力可 | `ai-node --mode company` |
| Private | 完全ローカル | 個人メモ/機密 | 一切共有しない。volatile短期TTLもここ | `ai-node --mode private` |

**UI原則**: 実行アイコンをモード別に分離(AI Public / AI Company / AI Private)。利用者が現在モードを常に意識でき、誤操作で機密がPublicへ漏れない。モードは推論時判定でなく**起動時選択**。

**[補完] 主戦場の段階展開(Company Phase1先行 → Public Phase2後続)**: オーナー採用決定(2026-07-17)により、上表のPublic/Company/Privateは同時並行整備ではなく**Company層をPhase1として先行実装し、Public層をPhase2として後続させる段階展開**を取る。両フェーズ共通の背骨(trait 3本/エントリ形式の不変コア/triples/volatility/共有ゲート/検索)はPhase1から確定・実装するが、Public層でのみ必須となる機構(witness+アンカー二層、評判+ステーク+外部照合裁定、revocation+regurgitation)はPhase2に繰り延べる。ただし省略は「削除」ではなく「未実装スロット」として扱い、エントリ形式には `witness_sigs` / `anchor_proof` / `stake` / `trust` をCompany Phase1の時点から空(optional)の可変状態フィールドとして型で確保する(**[補完]** うち `trust` の `independent_agreement`/`supporting_versions` 2フィールドのみ、S4層1先行実装〔2026-07-19〕によりPhase1で充填済み。→§8冒頭[補完]・§6差異表)。幻覚パリティ(§10関連)は概念(Tierタグ)をPhase1から導入するが、強制機構(Tier-H裁定)はPhase2から。アンカー二層・ステーク経済はPhase2に全繰り延べつつ、対応する可変フィールドの空スロットのみ先に確保する。詳細・採用経緯は[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.7、一次設計判断は[信頼性設計メモ.md](./信頼性設計メモ.md) §9、段階と既存S1〜S7ステージの対応は[Roadmap.md](./Roadmap.md)を参照。

## 10. 揮発性タグの付与・訂正・攻撃耐性

タグは単一主体が決める属性ではなく **`{class, confidence, evidence, last_reeval}` の確率的推定**(§6スキーマ)。証拠が集まるほど収束させ、§8の機構に相乗りする。

**[補完]** S2.5実装では、タグの運用値(`volatility_class_operative`/`confidence`/`evidence`)は署名対象外の mutable_state 側に分離され、coreには著者の初期主張 `initial_volatility_class` のみが固定される(→§6 [補完] 差異表)。また統治目標の再定義([信頼性設計メモ.md](./信頼性設計メモ.md) §9)の下では、本節の訂正・攻撃耐性は「真実の確立」ではなく「非劣化の維持(敵対的タグ操作を自然ノイズの水面下に戻す)」として機能させる。Tier-H領域はパリティ管轄外であり別機構(Phase2)で扱う(→§8.1 [補完])。

### 10.1 初期付与(知識グラフ分解が主軸、LLM自己申告は補助)

| 案 | 採否 | 根拠 |
|---|---|---|
| 案4 知識グラフ分解 | 主軸 | 述語の型で機械分類。「Xの開発者はY」=permanent、「Xの最新版は」=時間指示語ありでvolatile。述語オントロジーに揮発性を事前付与。主観がなく再現可能=攻撃者が動かしにくい |
| 案1 LLM自己申告 | 補助信号 | 信頼できない(モデル差/カットオフ自認不可/悪意ローカルLLMは嘘可)。有効なのは逆向き:「最新情報が要る/自信ない」申告はvolatile側へ倒す(安全側) |

**初期ルール(安全側に倒す)**
1. 分解成功かつpermanent型述語 → `permanent(confidence:中)`
2. 時間指示語を含む → 強制 `volatile`
3. 分解失敗/未知述語 → デフォルト `slow`
4. LLM自己申告は不一致時に確信度を下げる方向のみ

根拠=誤分類コストの非対称性。「volatile→permanent誤り(毒が居座る)」は「permanent→slow誤り(早めに再検証)」より遥かに危険。

### 10.2 訂正(投票でなく反証つき証拠で動かす)

- 層3抜き打ち再推論で既存回答と食い違えば確信度↓・再分類キューへ。slowはTTL満了で自動再評価
- 「volatileだ」の主張は矛盾する検証済み事実へのポインタ(反証)を伴って初めて効力。単なる★は無視
- **昇格は厳しく、降格は緩く**: slow→permanentは複数TTL周期の一致実績が必要。permanent→降格は反証1件で進む

### 10.3 攻撃耐性(非対称対応)

| 攻撃 | 対策 |
|---|---|
| A: volatileをpermanentと偽り毒を永続化 | 昇格に時間をまたいだ独立一致実績を必須化(偽装は独立性検証で露見)/ permanentでも抜き打ち検証を止めない(検証率を下げるだけ)/ 僭称露見で登録ノードを最大スラッシング |
| B: 正しいpermanentをvolatile化して消す(検閲) | 降格は「削除」でなく「再検証頻度の引き上げ」に留める→正しければ自動復帰/降格主張も反証必須、反証なき大量要求は局所EigenTrustで減衰/複数版併存でpermanent版は消えず消費側が選べる |

## 11. 法的リスク緩和のプロトコル要件

> 法的助言ではない。Public層公開前に専門弁護士レビュー必須。

**原則**: 「事実トリプルしか保存しない + 消せる + 誰が入れたか分かる」の3点をプロトコルに焼き込めば、Winnyとの法的距離を構造的に取れる。どれか一つでも後付け運用に回すと幇助論に接続されうる。

| 要件 | 実装 | 対応する脅威 |
|---|---|---|
| R1 トリプルのみ保存 | 長文逐語は保存せず事実トリプル+出典メタ。回答は受信側で毎回再合成。トリプル化不能(詩/コード/表)はPublic拒否 | 第三者著作物の混入(最も現実的なリスク) |
| R2 regurgitationフィルタ | n-gram+近傍検索で既存著作物との重複検知。結果もwitness署名対象。通過証明をprovenanceに必須記録 | 逐語再現=複製=侵害 |
| R3 失効(revocation) | 中央なしでも署名付き失効通知をフラッディング、各ノード自律配信停止、評判連動 | 「通知で止められる」=日米EU免責の入口。削除不能P2Pはセーフハーバー外 |
| R4 出所の完全記録 | 生成元Agent/時刻/author署名/witness/フィルタ通過証明を必須。**匿名性でなく説明可能性を売りに** | Winny判例で匿名性が最も不利に働いた |
| R5 Agent規約マトリクス | Public層は再配布明示許可ライセンスのモデルに限定。商用API出力はCompany/Privateに留める | 各社規約(競合モデル学習禁止条項)への抵触 |
| R6 ドキュメント衛生 | 適法用途の明記+AUP。開発者のグレー用途示唆を排除 | Grokster inducement / Winny幇助論 |
| R7 法人格分離・段階公開 | リファレンス実装は財団/OSS。フィルタと失効が実証されるまでPublic層は招待制/小規模 | 開発者個人を「運営者」に見せない |

**[補完]**(2026-07-19、R2/R3/R4先行設計ノート) R2(regurgitationフィルタ)/R3(revocation)/R4(出所記録)の3本柱について、Public Phase2専属の先行設計(骨子+論点構造化)を [S5_Public_Phase2_法的機構先行設計.md](./S5_Public_Phase2_法的機構先行設計.md) として追加した。実装未着手・S1〜S7の段階定義/ゲート/進捗ステータスは不変。**法的助言ではない。Public層公開前に専門弁護士レビュー必須**(本節冒頭の免責と同一)。

**[補完]**(2026-07-19、R5未整理事項) ローカル推論モデル(`gemma3` / `glm4` 等、Ollama経由)自体の利用規約・モデルライセンス(出力の再配布・共有条件)は、現行R5では**未整理**である(現行R5は「商用API出力はCompany/Privateに留める」という主旨のみで、ローカル推論モデルの規約は扱っていない)。Ollama経由エントリの他ノードとの共有(S3の多ノード共有パイプラインへの合流=Company層、および将来のPublic層共有)を本格化する前に、本マトリクスへ「ローカル推論モデル」行を追加し確認が必要。一次記載は[実装スペック(2026-07-18)](./superpowers/specs/2026-07-18-selectable-inference-backend-design.md)末尾の追記(2026-07-19)。**法的助言ではない。Public層公開前に専門弁護士レビュー必須**(本節冒頭の免責と同一であり、本補完はそれを薄めるものではない)。

- **既定は共有オフ(オプトイン)**: ソフトウェアは既定で「ローカル/Private」または「共有停止」で動作する。ネットワーク共有はユーザーが明示的に有効化(companyモード起動 + 共有on)して初めて働く。
- **いつでも即オフ(共有キルスイッチ)**: 共有を有効化した後も、再起動なしに即座に停止できる(CLI `--sharing off` / 実行中の `POST /v1/sharing`、`src/core/sync.rs` の `sharing_enabled`)。停止するとannounce/供出/Digest列挙/anti-entropy/取り込み起動の5経路が止まり、ローカルキャッシュ利用のみが残る(既存キャッシュの削除ではない)。
- **弁護士レビューの位置づけの明確化**: 「softwareが存在・動作するためのハードゲート」ではなく、「**Publicでの大規模共有ネットワークを運用する主体への推奨**」である。個人がprivate/共有停止で使う分には、このレビューの完了状態に縛られない。R7(法人格分離・段階公開)と整合する(リファレンス実装は財団/OSS、運用主体が別途リスクを負う)。
- この再定義は本節冒頭および直下の免責文言と**矛盾しない**。免責(「Public層公開前にレビュー必須」)は運用者への注意として存置し、本補完はそれを薄めるものではない。
- キルスイッチの停止保証の技術的スコープ(送出5経路 vs レジストリ・プレゼンス公告の扱い、`/v1/*`と`/wire/*`の境界)は [Roadmap.md](./Roadmap.md) §2 S6節の横断注記を参照。

## 1. 競合分析 — 「作る意味はあるか」

### 結論

**要素技術はすべて既存。ただし「ユーザー横断で共有する意味キャッシュ + ネットワーク物理分離」という組み合わせは空白地帯。ここに賭ける価値はある。**

### 要素ごとの既存プロジェクト

| この構想の要素 | 既存 | 成熟度 |
|---|---|---|
| Semantic Cache | GPTCache, Redis Semantic Cache | 商用化・枯れている |
| プライバシー配慮ローカルキャッシュ | MeanCache(連合学習) | 研究段階 |
| P2P分散推論 | LLMule, POKT Network, Petals | 実働あり |
| 回答の投票/検証 | LLM-Net, BC4LLM, Proof-of-Thought | 研究段階 |

### MeanCache との精密な差分

| 論点 | MeanCache | 本構想 |
|---|---|---|
| キャッシュ所在 | 各ユーザー端末ローカル | P2P分散共有 |
| 共有範囲 | **完全プライベート(共有なし)** | **人類規模で共有** ← 核心 |
| プライバシー手法 | 連合学習(技術で守る) | ネットワーク物理分離(Public/Company/Private) |
| Embeddingモデル | MPNet / Albert(768次元) | 未定 → MPNet級を推奨 |
| 類似度しきい値 | MPNet τ=0.83 / Albert τ=0.78 | 未定 → 共有用は0.9+推奨 |
| 文脈の扱い | context-chain をメタで保持・照合 | 未対策(穴) |
| 保存圧縮 | PCA 768→64次元、83%削減 | 未定 → 採用推奨 |
| Llama-2 | 実証済みで不適(0.04秒/F1=0.75/30GB) | 同様に回避すべき |

**借りるべきもの**: Embeddingモデル(MPNet級)、しきい値0.8前後、PCA圧縮、context-chain照合。
**発明すべき唯一の本丸**: ユーザー横断で共有されたキャッシュの信頼性(MeanCacheは共有しないため未解決)。

## 2. 共有キャッシュの信頼性 — 4つの脅威

| 脅威 | 内容 | 対策 |
|---|---|---|
| A. 誤ヒット | 意味しきい値が他人の別意図質問に当たる | 共有用τをローカルより高く(0.9+)、精度優先 |
| B. 文脈欠落 | 「色を赤に変えて」等、文脈で答えが変わる | 文脈依存質問は共有対象外。文脈自立質問だけ共有 |
| C. キャッシュ毒 | 悪意ある誤情報の注入(分散共有ネットワークの宿命) | 案2(出所ハッシュ)+案3(評判付き投票)+案4(事実分解) |
| D. 鮮度 | 答えが古くなる | 平文日付 + 揮発性タグ(TTL) |

**設計原則: 共有キャッシュに載せるのは「文脈自立 × 事実型」の質問だけ。** 時事・文脈依存はローカル or 都度推論。これで脅威B・Dを構造的に回避。

## 3. 鮮度設計 — 日付だけでは足りない

同じ作成日でも有効期間は質問ごとに違う(「Winnyの開発者」=永久 vs 「最新のClaude」=数ヶ月)。
→ 必要なのは **作成日 + 揮発性クラス(TTL)** のセット。

```text
volatility:
  permanent  : 事実・不変 → 日付無視で使える
  slow       : 数年 → 作成日+猶予で判定
  volatile   : 時事 → 短TTL、切れたら再推論
```

揮発性の判定は **案4(知識グラフ化=事実へ分解)** が担う。「開発者=X」は事実型=permanent と分類できる。

## 4. 詐称防止 — ハッシュではなく署名

### 重要な原則: ハッシュは「改ざん検知」であって「詐称防止」ではない

- ハッシュは公開関数 → 攻撃者も再計算できる → 日付を書き換えてハッシュも付け替え可能。
- 防げるのは「うっかり壊れた」まで。「悪意で書き換えた」は防げない。

### 詐称を防ぐ2つの道具

1. **デジタル署名**(誰が言ったかを固定)
   - 投稿ノードが秘密鍵で `署名(質問 + 回答 + 日付)`。検証は公開鍵で誰でも可、作成は本人のみ。
   - 穴: 投稿者本人が最初から嘘日付で署名したら防げない → 道具2で補う。

2. **信頼できるタイムスタンプ**(いつを第三者が保証)
   - P2Pには共通の時計がないため外部の証人が必要。
   - **中央の時計を持たない分散ネットワークなら「受信ノードの合意」が自然**: 複数ノードが「最初に受信した時刻」を署名付き記録。未来日詐称を弾ける。
   - 他選択肢: RFC 3161 TSA(中央依存)、ブロックチェーンのブロック高(重い)。

### エントリ最終形

```text
filename:  a3f9c2...              ← hash(全体)  ※改ざん"検知"用（案2、正しく採用）
中身:
  answer:       "開発者=金子勇"
  claimed_date: 2026-07-16        ← 投稿者主張の日付（平文・比較用）
  volatility:   permanent
  author_sig:   <投稿ノードの署名>  ← 道具1: 誰が言ったか固定
  witness_sigs: [                 ← 道具2: いつ受信したか複数ノードが保証
     {node: X, seen: 2026-07-16, sig},
     {node: Y, seen: 2026-07-16, sig}
  ]
```

## 5. 共有対象の線引き仕様 — 文脈自立 × 事実型

### 基本構造: 2軸のANDゲート + 保守的デフォルト

```text
共有可能 = 文脈自立(軸1) AND (事実型 かつ 揮発性∈{permanent, slow})(軸2)
どちらか落ちたら → ローカル限定(Private) or キャッシュしない
デフォルトは「共有しない」。全ゲート通過時のみ昇格。
```

**なぜ保守的デフォルトか**: 共有キャッシュは人類規模で汚染が広がる。
「共有すべきを取りこぼす(偽陰性)」より「共有すべきでないを混ぜる(偽陽性)」の方が遥かに有害。
→ 精度優先。迷ったら共有しない。

### 判定はいつ行うか

キャッシュ登録は「ミス後、Agent回答後」に起きる。判定は**書き込み時に1回だけ**。

| 経路 | タイミング | やること | コスト |
|---|---|---|---|
| 読み取り(hot) | 毎クエリ | 共有キャッシュを引くだけ。重い判定なし | 極小 |
| 書き込み(登録時) | ミス+回答後 | フル判定して共有フラグを決める | 許容 |

### 軸1: 文脈自立の判定(安い順)

| 層 | 手法 | 除外例 |
|---|---|---|
| L0 語彙 | 指示語・代名詞の裸出現 | それ/その/これ/あれ/さっきの/上記の/彼, it/that/above |
| L0 省略・命令 | 対象省略の命令形 | 赤に変えて / もっと短く / 続けて / 次は? |
| L2 Agent自己申告 | 登録時に1問 | 「前提の会話なしで単独回答できるか? yes/no」 |

### 軸2: 事実型/揮発性の判定

| 層 | 手法 | 除外するもの |
|---|---|---|
| L0 揮発語彙 | 時事トリガー | 最新/現在/今/今日/価格/株価/天気/2026年 → volatile |
| L0 主観語彙 | 意見・主観 | おすすめ/べき/どう思う/best/should |
| L0 個人参照 | 一人称・自分データ | 私の/自分の/このファイル |
| L2 Agent + 案4 | 回答を事実トリプルへ分解 | 分解できれば factual/permanent。生成文は除外 |

揮発性は permanent / slow / volatile をタグ付けし、**permanent と slow だけ共有**。volatile はローカル短期TTLのみ。

### 判定フロー(登録時)

```text
回答生成完了
  │
  ▼
[L0 ルール] 文脈依存語・揮発語・主観語・個人参照 を含むか?
  │ 含む → 共有しない（大半をここで高精度に足切り）
  │ 含まない
  ▼
[L2 Agent] 単独回答可能? 事実型? volatility=?
  │ どれか No / volatile → 共有しない
  │ 全部 Yes かつ permanent|slow
  ▼
共有キャッシュへ昇格（署名+witness付きで登録）
```

L0 で大量を高精度に足切り → 生き残りだけ L2 で精査。NPU/GPUをほぼ使わずに大半判定でき、省電力思想と一致。

### 具体例

| 質問 | 軸1 | 軸2 | 判定 |
|---|---|---|---|
| Winnyの開発者は? | 自立 | 事実・permanent | ✅ 共有 |
| P2Pの仕組みを教えて | 自立 | 事実・slow | ✅ 共有 |
| 最新のClaudeは? | 自立 | 時事・volatile | ❌ ローカル短期 |
| おすすめのエディタは? | 自立 | 主観 | ❌ 共有しない |
| 色を赤に変えて | 依存 | — | ❌ 共有しない |
| このコードのバグは? | 個人参照 | — | ❌ ローカルのみ |

### キャッシュ適格性の層設計 — 共有ゲートは唯一の門番ではない【2026-07-19 追記】

> 出典: 設計議論(オーナー×Claude×fableレビュー、2026-07)。本項がこの層設計の一次記載であり、[Architecture.md](./Architecture.md) §7 の [補完] はここを参照する。

本節の共有ゲート(文脈自立 AND 事実型 AND 非volatile)は**ネットワーク共有の門番**であって、**ローカルキャッシュへの登録・ヒットの門番ではない**。「共有できるか」と「キャッシュしてよいか」を混同しないため、キャッシュ適格性を次の多層として定義する(入れ子: 共有適格 ⊂ 永続ローカル適格。その外側はセッションスコープに閉じる)。

| 層 | 条件 | 強制点 | 実装状況(2026-07-19) |
|---|---|---|---|
| 共有適格(既存・最も狭い) | 本節の共有ゲート(L0)+ `judge_entry` の全段ANDゲート | 登録時に1回(本節「判定はいつ行うか」)+受信側でロード時に再導出(S2.5) | 実装済み(S2/S2.5/S3) |
| 永続ローカル適格 | **情報要求型 AND 非個人** | TTL・クラス別しきい値を**登録時ゲートでなく lookup(検索)時に強制** | 登録ゲート(情報要求型判定)は未実装 — 現行実装は全回答を無条件にローカル登録し共有のみゲートする。lookup時強制は部分実装(→下記「既知の穴」) |
| セッション(会話)スコープ | マルチターン・文脈依存・雑談はここに閉じる | 会話終了で破棄 | 未実装(将来拡張) |

**原則(キャッシュ可能性の本質条件)**: キャッシュ可能性の本質条件は「**回答が質問の安定な関数であること**」である。本節の主観/客観の区別はその**不完全な代理変数**にすぎない。切るべき軸は主観性ではなく**発話行為タイプ**(情報要求 vs 挨拶・対話行為・生成依頼)である。例:「調子どう?」は本節のどの語彙表(文脈依存語・主観語・個人参照・揮発語)にも当たらないが、情報要求ではない(回答が質問の安定な関数でない)ためキャッシュ不適格。

**Phase1の現実解(段階)**: セッション層を今すぐ作る必要はない。「雑談・文脈依存発話は**そもそもキャッシュ登録しない(記録もしない)**」という二層構成でも越境事故(過去会話由来の定型文が別文脈でヒットする等)は防げる。セッション層は必要になってから足す。

**補足事実(誤解防止)**: S2.5形式は answer 平文を保存しない(core=factsのみ)。雑談は facts が空になるため、**現構造では「古い定型文がヒットして返る」事故はそもそも起きない**。ただし将来ローカルに answer 平文保存を再導入する設計変更をする場合、それは S2.5 が著作権面・毒面積の縮小(§8、[Architecture.md](./Architecture.md) §11 R1・非目標N5)のために捨てたものの**部分復活**である、と自覚的に扱うこと。

**設計原則(強制点の置き場)**: 登録時ゲートは情報を**不可逆に**捨てる。これに対し TTL・クラス別しきい値は **lookup(検索)時に強制**すべきである — 検索時ゲートは後からチューニング可能で、判断の変更が既存エントリにも遡って効く。S3実装が採った「TTL満了は検索から除外し物理削除しない」方式([S3_Company_Phase1_社内多ノード共有設計.md](./S3_Company_Phase1_社内多ノード共有設計.md) §7)はこの原則の実装形である。lookup時強制の現状の穴(poc/はTTL機構なし・src/coreはクラス別τ未配線)は [Architecture.md](./Architecture.md) §7 [補完]「既知の穴」・[Roadmap.md](./Roadmap.md) §3 に記録した。**追記(2026-07-20)**: このうち src/core の「クラス別τ未配線」は、**共有由来(他ノードから受信した)エントリへの `SHARED_THRESHOLD`(0.90)のlookup時配線により解消済み**(根拠は本メモ §2 脅威A「他人の別意図質問への誤ヒット」=共有側は精度優先。判別は `MutableState.origin_received`、既定`true`=保守側・署名対象外・ノードローカルstateのみ。→ Architecture §7 [補完]「既知の穴」)。TTLの確定値チューニングは引き続き実測待ち([Roadmap.md](./Roadmap.md) §3)。

### 手続き的知識(プログラミング一般知識・コード)— S4前提の将来レーン【2026-07-19 追記】

> 出典: 設計議論(オーナー×Claude×fableレビュー、2026-07)。本項が一次記載。依存関係・進捗は [Roadmap.md](./Roadmap.md) §2 S4節・§3 を参照。

| 論点 | 判断 | 根拠 |
|---|---|---|
| 現状ゲートとの関係 | 現状のトリプル(s,p,o)ベース共有ゲート(本節+案4)は**宣言的事実**向けであり、手続き的知識(手順・コード)は分解できず facts が空になって共有除外される。これは正しい挙動であり、**「ゲートに例外を掘る」話ではない** | 案4(§3・§5軸2)の適用範囲そのもの |
| なぜ独立した設計フェーズか | 手続き的知識の共有は answer/コード本文の**保存・伝播を要する第二のエントリ形式**であり、S2.5の核心設計(core=factsのみ=著作権面・毒面積の最小化。§8、Architecture §11 R1/非目標N5)を破る | 「トリプル化できないものはPublic層に載せない」(§8)との整合を壊さずに扱うには別形式が必要 |
| 位置づけ(本命根拠) | プログラミング一般知識は長期的に有力ユースケース。ただし本命根拠は「需要」ではなく**機械検証可能性** — コンパイル・実行テスト・lintが効く唯一の領域で、S4内在信頼度・witness再検証(=N-of-Mサンドボックス実行)と最も相性が良い。したがって**S4なしに着手してはならない** | 独立検証(§6層1)を「学習データの合意」でなく機械的検証に接地できる唯一の領域(§9 統治目標の再定義と整合) |
| 陳腐化 | コードは時間でなく**リリースイベント**で腐る。時間ベースvolatility/TTL(§3)ではなく、エコシステム/ライブラリ/semver範囲を**署名対象メタデータ**に含め、リリースフィード駆動で失効させる設計が正道。バージョン無指定の質問(暗黙に最新仮定)は**共有不可・ローカル限定**が安全線 | §3の時間TTLモデルは宣言的事実向けで、コードの失効トリガ(リリース)と一致しない |
| 毒 | コードは**Tier-Lに置かず、Tier-H相当か専用ティア**とし、共有前にwitnessの機械検証を必須化する。コードの誤りは二値でなく段階的(動くが脆弱、等)なので、二値shareableフラグでなく**検証証跡付き信頼度**が要る(=S4の題材)。悪意あるパッケージ名混入(**slopsquatting型**)はコピペ実行文化と直結する実在の攻撃クラス | §2脅威C・§9 2ティア分類(低頻度・高被害はTier-H)と整合 |
| 社内パイロット | S3がCompany Phase1(社内限定)であることを活かし、毒脅威が組織的に緩和される**社内限定で手続き的レーンを先行実験**する段取りが検討に値する | §9(Company先行の8軸優位)の応用 |

## 6. シビル攻撃対策 — ノード評判設計

> Fable(claude-fable-5)による設計を採用。

### 核心の判断: 投票を「真偽の決定」から「探索の優先度ヒント」へ格下げ

シビル攻撃は「多数決」を攻撃するもの。ならば真偽を多数決で決めない。
**既存の「事実型(permanent/slow)だけ共有」という線引きが、そのままシビル対策の土台になる** —
事実の正しさは投票ではなく独立検証で決められるため。
※ permissionlessなシステムでシビルをコストゼロで完全排除するのは理論上不可能(Douceur 2002)。目標は「割に合わなくする」。

### 推奨アーキテクチャ: 検証主軸の3層

```text
層1 エントリ内在の信頼度(シビル非依存)★主軸
  独立生成された複数回答の「事実分解トリプルの一致率」（案4）
  + 出所ハッシュ / witness署名の時系列で"独立性"を検証
層2 ノード評判(局所EigenTrust)
  誕生証明(witness署名を流用) + ID生成PoW
  検証済み貢献を対数重みで累積 / 誤情報でスラッシング(評判焼失)
  グローバル合意はせず各ノードの近傍でのみ信頼を伝播
層3 消費側の最終防衛
  高信頼キャッシュでも確率的に抜き打ち再推論 → 層1/層2へフィードバック
```

### シビルを割に合わなくする: コストの掛け算

| コスト | 効き方 |
|---|---|
| ID生成PoW | 限界コストを非ゼロに(補助のみ。計算力は金で買えるので単体不十分) |
| 時間（最重要） | 新規IDは重みゼロ開始。1万ID作れても半年分の正しい貢献履歴は持たせられない。時間は並列化不可 |
| 貢献 | 重みを得る唯一の道が「後に正しいと検証される回答の登録」→ 攻撃準備がネットワーク貢献になる(tit-for-tat哲学) |
| スラッシング | 稼いだ評判は毒注入1回で焼失。長期投資を1回で失う経済的抑止 |

### 投票の重み付け(二次防衛)

- 票の重み = `f(ID年齢, 検証済み貢献, ローカル信頼)`、対数(上に凸)にして単一大物ノードの支配も防ぐ
- 否定票(反証つき)は肯定票より強く効かせる = 毒は「人気で埋める」より「反証で殺す」方がシビル耐性が高い
- 票自体を希少化(投票へのPoWスタンプ / 評判比例の投票上限)

### 既存技術の適否

| 技術 | 適否 | 理由 |
|---|---|---|
| EigenTrust | ○ 部分採用 | 「信頼を評価者の信頼で割り引き伝播」の核心のみ。pre-trusted peers(実質中央)は誕生証明+貢献履歴で置換。グローバル収束は局所計算に落とす |
| PoW(ID生成時) | △ 補助 | 限界コストを課すのみ。計算力は買えるため根本対策にならない。継続的PoWは電力浪費で不適 |
| Proof-of-Stake | ✕ 不向き | ステーク資産がない。トークン導入は投機・規制・匿名参加の思想と根本衝突 |
| Web-of-Trust | △ 補助 | ローカル信頼の初期ブートストラップに有効。手動署名依存でスケールせず主軸不可 |
| SybilGuard/Limit/Rank | △ 条件付き | 社会グラフ前提。匿名P2Pに自然な社会グラフはないが「長期取引履歴」を擬似社会グラフに流用可 |
| ブロックチェーン全順序合意 | ✕ 過剰 | キャッシュは結果整合+複数バージョン併存で十分。合意コストだけ輸入することになる |

### トレードオフ(明示)

- コールドスタート: 善意の新規も数週間は無力 → 既存ノードの紹介(紹介者を連帯担保)で緩和
- 抜き打ち再検証: ヒット率を数%犠牲。信頼度に応じ検証率を動的に下げる
- slow型の更新: 独立一致率は「古い定説」を過剰強化 → witness時刻で鮮度重み付けし、新反証が古い多数を覆せる経路を残す
- 局所評判の非一貫性: 「ノードAの評判」はノードごとに異なる(仕様。分散型ネットワークらしい緩さと整合)
- 国家級APT: 数年正直に振る舞う攻撃者は排除不能。層3の抜き打ち検証が最後の砦(=「完璧でなくてよい」の意味)

## 7. 揮発性タグを誰が付けるか

> Fable(claude-fable-5)による設計を採用。

### 核心の判断: タグは「単一主体が決める属性」ではなく「証拠で更新される確率的推定」

誰か一人(Agent / 知識グラフ / ネットワーク)に決めさせると、その主体が攻撃点になる。
タグを **`{推定クラス, 確信度, 根拠, 最終再評価時刻}`** の状態オブジェクトにし、証拠が集まるほど収束させる。
→ §6の3層評判・独立検証・複数版併存と同一機構に載る。

### フェーズ1: 初期付与（知識グラフ分解が主軸、LLM自己申告は補助）

| 案 | 適否 | 根拠 |
|---|---|---|
| 案4 知識グラフ分解 | ◎ 主軸 | 述語の型で機械分類できる。「Xの開発者はY」「Xの定義はY」=permanent、「Xの最新版は」=時間指示語ありでvolatile。述語オントロジーに揮発性を事前付与すれば分解成功時点でほぼ自動導出。主観がなく再現可能=攻撃者が動かしにくい |
| 案1 LLM自己申告 | ○ 補助信号のみ | 信頼できない(モデル差・カットオフ自認不可・悪意ローカルLLMは嘘可)。有効なのは逆向き:「最新情報が要る/自信がない」申告はvolatile側に倒す(安全側) |

初期ルール(安全側に倒す):
1. 分解成功 かつ permanent型述語 → `permanent(確信度:中)`
2. 時間指示語(最新/現在/今/latest)を含む → 強制 `volatile`
3. 分解失敗 / 未知述語 → デフォルト `slow`(permanentは強すぎ、volatileは情報を捨てすぎ)
4. LLM自己申告は不一致なら確信度を下げる方向にのみ作用

根拠 = 誤分類コストの非対称性。「volatileをpermanentと誤る(毒が居座る)」方が「permanentをslowと誤る(早めに再検証)」より遥かに危険。疑わしきはslow/volatile側へ。

### フェーズ2: 訂正（「投票」でなく「反証つき証拠」で動かす）

- 層3の抜き打ち再推論で既存回答と食い違えば確信度を下げ再分類キューへ。slowはTTL満了で自動再評価。
- 「volatileだ」という主張は矛盾する検証済み事実へのポインタ(反証)を伴って初めて効力を持つ。単なる★は無視 → 訂正経路自体のシビル耐性を担保。
- 昇格は厳しく、降格は緩く: slow→permanent昇格は複数TTL周期の一致実績が必要。permanent→降格は反証1件で進む。
- witness時刻で「いつから同じ回答が安定しているか」を観測 → タグの正しさを主張でなく観測履歴で裏付ける。

### フェーズ3: 攻撃耐性(非対称対応)

**攻撃A: volatileをpermanentと偽り毒を永続化**
- permanent昇格に時間をまたいだ独立一致の実績を必須化(偽装は独立性検証で露見)
- permanentでも抜き打ち再検証を完全には止めない(検証率を下げるだけ)= 毒永続化の最後の砦
- タグ僭称が崩れたら登録ノードを最大スラッシング(最も高価値な攻撃だから)

**攻撃B: 正しいpermanentをvolatile扱いにして消す(検閲)**
- 降格は「削除」でなく「再検証頻度の引き上げ」に留める → 正しければ自動復帰、可用性を落とさない
- 降格主張も反証必須。反証なき大量volatile化要求は局所EigenTrustで減衰
- 複数バージョン併存でタグも版ごと。permanent版は消えず、消費側が高信頼版を選べる

## 8. 著作権 / 責任の所在

> Fable(claude-fable-5)による設計を採用。※法的助言ではなく設計論点の整理。実装前にIT・著作権専門の弁護士レビュー必須。

### 一言でいうと

**「事実トリプルしか保存しない + 消せる + 誰が入れたか分かる」の3点をプロトコル自体に焼き込めば、Winnyとの法的距離は構造的に取れる。**
どれか一つでも「後付けの運用対応」に回すとWinny型の幇助論に接続されうる。

### 5論点

**1. LLM生成物の著作権 — 事実型限定が最大の防御壁**
- 純粋AI生成物に著作権は発生しないのが日米EUの主流。事実そのものは万国で著作権対象外。
- 但し著作権ゼロ ≠ 再配布自由。各社の利用規約(契約)が別レイヤーで拘束。特に「競合モデルの学習・開発への利用禁止」条項に、共有キャッシュが「他LLMの推論を代替する」構造が触れると主張されうるグレー = 規約上の最大論点。

**2. 第三者著作物の混入 — 最も現実的なリスク**
- LLMが記事・書籍・GPLコードを実質再現(regurgitation)すると、AI生成か否かに関係なく複製物=侵害になりうる。日本法の送信可能化権は広く、全キャッシュ保持ノードが送信主体になりうる。
- 知識グラフ分解(案4)が質的に効く: 事実トリプル+再合成なら保存されるのは表現でなく情報。限界: 詩・コード・表など事実と表現が不可分なものはトリプル化不能 → トリプル化できないものはPublic層に載せない。

**3. 開発者/ノード運用者の責任(Winny判例 2011)**
- 有利: 中央管理者なし / 事実型限定+混入フィルタ(Winnyは実態9割超が侵害、本件は設計で構造排除) / モード分離(Public層のみ再配布)。
- 危険: ⚠️匿名性を売りにする(Winnyで最も不利に働いた)→ 署名+評判でトレーサビリティを高め、それを前面に出す。⚠️削除手段ゼロの設計。⚠️開発者のグレー用途を示唆する発言・ドキュメント(公開ドキュメントの書き方自体がリスク管理対象)。

**4. 各国法制の差**
- 日米EU共通で「通知を受けたら止められる仕組み」が免責の入口。完全に削除不能なP2Pは日米EUすべてでセーフハーバーの外。米DMCAは指定代理人登録+repeat infringer policyが要件。

**5. 推奨設計(構造的リスク低減)**
1. 保存しないものを規定化: 長文逐語は保存せず事実トリプル+出典メタのみ、回答文は受信側で毎回再合成。トリプル化不能はPublic拒否
2. プロトコルレベルのテイクダウン: 中央なしでも署名付き失効通知(revocation)をフラッディングし各ノードが自律配信停止。評判連動
3. 出所の完全記録: 生成元Agent/時刻/署名/witness/regurgitationフィルタ通過証明を必須化。匿名性でなく説明可能性を売りに
4. Agent選定の規約マトリクス: Public層は再配布を明示許可するライセンスのモデルに限定。商用API出力はCompany/Private(非再配布)に留める
   - 追記(2026-07-19): ローカル推論モデル(Ollama経由の`gemma3`/`glm4`等)の利用規約・モデルライセンス(出力の再配布・共有条件)は本マトリクスで未整理。Ollama経由エントリの多ノード共有を本格化する前に「ローカル推論モデル」行の追加・確認が必要([Architecture.md](./Architecture.md) §11 R5の[補完]と同旨。法的助言ではない)
5. ドキュメント衛生: 適法用途の明記+AUP(Grokster inducement対策)
6. 法人格の分離: リファレンス実装は財団/OSS、開発者個人を「運営者」に見せない
7. 段階的公開: フィルタと失効機構が実証されるまでPublic層は招待制/小規模

### 既存設計へ追加された要件

- 署名付き失効通知(revocation)のフラッディング機構
- regurgitationフィルタ(n-gram/近傍検索で既存著作物との重複検知、結果もwitness署名対象)
- 長文を保存せず「事実トリプル+出典メタ」only、回答は受信側で再合成

**追記(2026-07-20、法的姿勢の現実的再定義・共有キルスイッチ)**: 冒頭の免責(法的助言ではなく設計論点の整理。実装前にIT・著作権専門の弁護士レビュー必須)を前提として維持したまま、個人開発者/リファレンス配布という実態に即し、姿勢を次のように明確化する(S1〜S7の段階定義・ゲートは変更しない。一次記載は実装スペック[2026-07-20-sharing-killswitch-and-legal-posture-design.md](./superpowers/specs/2026-07-20-sharing-killswitch-and-legal-posture-design.md))。
- 既定は共有オフ(オプトイン): ソフトウェアは既定で「ローカル/Private」または「共有停止」で動作する。ネットワーク共有はユーザーが明示的に有効化(companyモード起動+共有on)して初めて働く。
- いつでも即オフ(共有キルスイッチ): 共有有効化後も再起動なしに即座に停止できる(CLI `--sharing off` / 実行中の `POST /v1/sharing`、`src/core/sync.rs` の `sharing_enabled`)。停止するとannounce/供出/Digest列挙/anti-entropy/取り込み起動の5経路が止まり、ローカルキャッシュ利用のみが残る(既存キャッシュの削除ではない)。
- 弁護士レビューの位置づけの明確化: 「softwareが存在・動作するためのハードゲート」ではなく、「Publicでの大規模共有ネットワークを運用する主体への推奨」へ格下げする。個人がprivate/共有停止で使う分にはこのレビューの完了状態に縛られない(R7・法人格分離・段階公開と整合)。
- この再定義は冒頭免責と**矛盾しない**(免責は運用者への注意として存置。本追記はそれを薄めるものではない)。
- キルスイッチの停止保証の技術的スコープ(送出5経路 vs レジストリのプレゼンス公告の扱い等)は[Roadmap.md](./Roadmap.md) §2 S6節の横断注記を参照。

## 9. 主戦場の段階展開(Company先行 → Public後続)【オーナー採用 2026-07-17】

> 出典: [設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.7(オーナー採用決定の経緯・Public/Companyの8軸比較の詳細はそちらを参照)。本節がこの決定の一次記載であり、[Architecture.md](./Architecture.md) §9 の [補完] 段落はここを参照する。

### 決定

Public層(permissionless・匿名多数)の困難(シビル・毒・法務・インセンティブ・ヒット率)のほぼ全ては「誰でも参加できる」「参加者が匿名多数」という2条件から生じる。Company層はこの2条件を両方消すため、価値提案・ヒット率・脅威耐性・著作権/プライバシー/法務・インセンティブ・既存資産の再利用・製品化までの時間・上流陳腐化の全軸でCompany層が優位である。

採用する戦略は **Company層をPhase1として先行させ、Public層はPhase2として後続させる段階展開**。Company層で十分と判断された時点で止めることも正当な経営判断であり「逃げ」ではない。逃げに転落するのは、(a)「Publicをやると言い続けながら着手しない」場合、(b)「Company都合でエントリ形式の空スロットを空け忘れ、Public移行が作り直しに化ける」場合の2経路のみ。

### 共有コア(背骨)と fork必須の境界

両フェーズ共通(背骨): trait 3本(Embedder/Signer/Agent) / エントリ形式の不変コア / triples(事実トリプル分解) / volatility(揮発性タグ) / 共有ゲート(Tierフィールドは空で用意) / 検索。

Phase2でのみ実装するが「場所」は今から確保する(fork必須・空スロット方式): witness+アンカー二層 / 評判+ステーク+裁定 / revocation+regurgitation(§8参照)。エントリ形式には `witness_sigs` / `anchor_proof` / `stake` / `trust` を、Company Phase1では空(optional)の可変状態フィールドとして型で確保する。省略は「削除」ではなく「未実装スロット」として扱う。**追記(2026-07-19)**: このうち `trust` の層1の2フィールド(`independent_agreement`/`supporting_versions`)のみS4層1先行実装によりPhase1で充填済み([Architecture.md](./Architecture.md) §8冒頭[補完]・[Roadmap.md](./Roadmap.md) §0対応表S4行が認めた例外。ただし案Bにより `state.json` へは非永続化=起動時にローカル版集合から再導出される導出状態であり、他の空スロットが空のままである原則は不変)。

### 幻覚パリティ・アンカー/ステークの仕分け

- 幻覚パリティ(2ティア+テール除外): 概念(Tier分類)はPhase1から導入可能、強制機構(Tier-H裁定)はPhase2から。定義・一次記載は次項「統治目標の再定義」。
- アンカー二層・ステーク経済: Phase2に全繰り延べ。ただしエントリ形式に `anchor_proof` / `stake` の可変フィールドを空で確保しておく(ネイティブトークンではなく評判ベースのステークであり、非目標N6のトークン経済圏排除とは矛盾しない)。

### 統治目標の再定義: 幻覚パリティ + テール除外【一次記載】

> 典拠: [設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.5(条件付き採用の判定)・§4.7(段階帰属)。本項がこの統治目標の一次記載であり、[Architecture.md](./Architecture.md) §7/§8.1/§10/§12 の [補完] はここを参照する。

**再定義**: 共有キャッシュの統治目標を「真実の保証」から「**幻覚パリティ + テール除外**」へ再定義する。達成目標は「共有キャッシュ経由の誤答率を、スタンドアロン(生)LLMの同クラス幻覚率以下に保つ」という**非劣化目標**であり、真実そのものの保証ではない。

| 論点 | 判断 | 根拠 |
|---|---|---|
| なぜ「真実の保証」を降ろすか | 達成不能なため。主要LLMは学習コーパスを共有し同じ誤りに独立到達するので、独立検証のトリプル一致率が測るのは「真実」ではなく「学習データの合意」 | 設計レビュー §1.3・§4.5 |
| 正しい理由付け | 「みんな幻覚を承知だから」ではなく「真実の保証が不可能だから、達成可能な非劣化目標へ切り替える」。ユーザーが受容しているのは自分が引いた確率的クジであり、他人が仕込んだ狙い撃ちではない(故意注入はパリティの外) | 設計レビュー §4.5(理由付けの罠) |
| 採用条件(3つ全て必須) | (a) 評価はクラス別採点(全体平均で薄めない)/(b) 故意注入はパリティ管轄外(ステーク+スラッシング+失効で対処)/(c) テール(低頻度・高被害)は別レイヤで守る | 設計レビュー §4.5(最終判定) |
| 既存機構への波及 | 独立検証・投票・witness等は「真実確立」ではなく「敵対的寄与を自然ノイズの水面下に戻す(不自然な一致・増幅の検出)」機構へサイズダウンして読む | 設計レビュー §4.5(屋根の効用) |

**2ティア分類**:

| ティア | 対象 | 目標・機構 |
|---|---|---|
| Tier-L(低リスク・パリティ適用) | 一般常識・OSS仕様・歴史的事実・定義(permanent事実の大半) | クラス別幻覚パリティ(このクラスの共有誤答率 ≦ 生LLMの同クラス幻覚率)。機構は軽量: 共有ゲート+揮発性+独立検証を「敵対シグナル検出」にサイズダウン |
| Tier-H(高リスク・パリティ禁止) | 医療・法律・金融・安全、特定個人/法人を主語に持つトリプル、および低頻度であること自体(統計が立たない=守れない) | パリティ不適用。ステーク+外部権威照合裁定+失効をフル適用、またはそもそも共有しない。「疑わしきは共有しない」を最大限保守側へ寄せる |

**段階帰属**(→本§9冒頭・[Roadmap.md](./Roadmap.md) §0): 概念(Tierタグ・分類)はCompany Phase1から導入する。強制機構(Tier-H裁定・外部照合)はPublic Phase2へ繰り延べる。

**実装状況(2026-07-17、S2.5)**: `initial_tier`(0=Tier-L / 1=Tier-H)は immutable_core の署名対象である — これは「真として信頼させる」ためではなく、著者の初期主張を固定して事後説明責任(スラッシング根拠)を成立させるため。運用値 `tier_operative` は受信側がロード時に再導出し、送信者の値は信頼しない([S2.5_エントリ形式設計.md](./S2.5_エントリ形式設計.md) §1・§6手順8)。既定値は現状 Tier-L(ティア分類器未実装)であり、tierを共有ゲートへ配線する段階で「未分類は Tier-H(疑わしきは保守側)」へ反転させる(同ノート§13 Medium-1)。

### 個別機構は依然未採用

段階展開そのものは採用決定だが、内側の個別機構(アンカー二層の具体構成/ステーク+外部照合裁定の具体設計/witnessの具体設計/Tier-H裁定の具体設計)は依然「未採用の候補」であり、Public Phase2で改めて判断する。詳細検討は[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.2〜§4.5、進捗管理は[Roadmap.md](./Roadmap.md)を参照。

### プロダクトの性格づけ — 「雑学」と「共有知識」は二択でなく段階【2026-07-19 追記】

> 出典: 設計議論(オーナー×Claude×fableレビュー、2026-07)。本項が一次記載であり、[Architecture.md](./Architecture.md) §1 の [補完] はここを参照する。

「雑学キャッシュ」に留めるか「共有知識キャッシュ」を狙うかは**二択ではなく段階**である。

| 論点 | 判断 | 根拠 |
|---|---|---|
| 雑学(宣言的事実)の位置づけ | 信頼機構の**訓練場**として最適。ただし**ゴールではない** | 失敗が良性(Tier-L幻覚パリティの管轄内)で機構を安全に鍛えられる一方、推論コストの低下で雑学キャッシュ単体の経済価値は侵食されていく |
| この系の本当の資産 | 雑学の中身ではなく、**署名・再導出・揮発性・信頼度という検証インフラ**そのもの | エントリの中身は陳腐化するが、検証インフラは対象領域を替えて再利用できる(手続き的知識レーン等。→§5追記) |
| 正確な性格づけ | 「**共有知識キャッシュを目指すが、信頼機構(S4以降)の成熟度がカバー範囲を規定する**」 | 上2行の帰結。カバー範囲の拡張は機構の成熟に従属する段階であって、宣言で決まる二択ではない |
| 雑談とコードの区別(混同禁止) | **雑談=原理的にキャッシュ不適格**(回答が質問の安定な関数でない → スコープ設計で恒久除外。→§5追記の層設計)。**コード=適格だが現信頼機構では時期尚早**(S4待ち。→§5追記の手続き的知識レーン) | 除外理由が異なる: 前者は本質条件の不成立(恒久)、後者は機構の未成熟(段階) |
| 依存関係 | **S4成果物(内在信頼度・検証証跡)が手続き的知識レーンの前提条件** | [Roadmap.md](./Roadmap.md) §2 S4節に明記 |

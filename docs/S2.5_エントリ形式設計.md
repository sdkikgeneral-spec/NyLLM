## 0. 全体を貫く決定

**authoritative(署名・ハッシュ対象)= serde 非依存の長さ接頭辞バイナリ(= immutable_core)。on-disk(保存・パース)= serde JSON エンベロープ。**

serde JSON エンベロープは一度もハッシュしないため、serde の内部表現(`BTreeMap` キーソート等)に一切依存しない。これにより [設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §2.2 弱点2(正準化が serde_json の内部実装依存で脆い — `preserve_order` feature が推移的に有効化された場合の全エントリ無効化リスク、C#/Rust 間のバイト一致非保証、Unicode 正規化の欠落)を、「禁止をドキュメントで守る」設計から「そもそも認証経路に serde が存在しないため原理的に発生しない」設計へと転換する。

## 0.1 レイヤ帰属(Public要素はどこに入るか)

この形式には Public層(Phase2)向けの要素が既に含まれているが、あくまで **「署名されない空スロット」としてのみ** であり、`entry_id` には一切影響しない。

- **immutable_core(署名・entry_id対象)= 全てレイヤ中立**。Company/Public共通の背骨であり、Public専用要素は1つも含めない。
  - 紛らわしい点として、`initial_tier`(幻覚パリティのティア、[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.5)は core に含まれるが、これは Public専用ではない。ティア分類は Company Phase1 でも使用する(社内でも医療・個人情報は Tier-H 相当のタグを要する)ため、レイヤ中立として扱う。
- **mutable_state の Phase2空スロット = Public要素**: `trust` / `witness_sigs` / `anchor_proof` / `stake` の4つ。いずれも署名対象外・`entry_id` 対象外であり、Phase1 では常に空(`None` または空 `Vec`)。Phase1 の動作はこれらを読み書きも参照もしない。
- **今「場所だけ」予約する理由**(段階展開の核心。[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.7「二律背反の捌き方」参照): 中身は Phase2 まで書かないが、「可変状態側・署名対象外」という配置と署名境界だけを今確定する。これを行わずに後から `witness`/`anchor`/`stake` を足すと署名境界が動き、全 `entry_id` が変わってしまう(=作り直し。[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.6「動く的撃ち」の罠に相当)。境界を今引いておけば、Phase2 は空スロットの中身を埋めるだけで済み、`entry_id` 不変のまま「追加」として完結する。逆に「Company用の純粋形式を先に作り、Public要素は後で別途設計する」という進め方は、この作り直しリスクを踏む。よって最小限の予約(境界のみ確定・中身は空)に留める。

## 1. immutable_core の全フィールド

署名対象・ハッシュ対象・作成後不変。`entry_id = sha256(正準バイト列)`。浮動小数点数を1つも含まない(埋め込みベクトルやロジットのような非決定的な数値をハッシュ鍵にしないという原則、[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §2.3 補足に対応)。

| # | フィールド | 型 | 意味 | 正準表現 |
|---|---|---|---|---|
| 1 | `schema_ver` | `u16` | スキーマ版(現行=1) | 2バイト big-endian |
| 2 | `question_norm` | `string` | NFC正規化・trim済み質問文(表示・再embedding用) | `lp_str` |
| 3 | `facts` | `array<FactTriple>` | 事実トリプル。`triples.rs::FactTriple{s,p,o}` と同一。`decompose` の決定的順序を保持 | `u32 count` + 各 `lp_str(s) lp_str(p) lp_str(o)` |
| 4 | `provenance.agent` | `string` | 生成Agent名(署名対象化。[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §2.2 弱点1・[Roadmap.md](./Roadmap.md) 残存リスクcへの対応) | `lp_str` |
| 5 | `provenance.model` | `string` | モデル識別子(モック時 `""`) | `lp_str` |
| 6 | `provenance.embedder_model_id` | `string` | 著者が使った embedder 識別子(情報用) | `lp_str` |
| 7 | `created` | `string` | UTC RFC3339・秒精度・Z終端 | `lp_str` |
| 8 | `initial_volatility_class` | `string` | 著者の初期揮発性クラス `permanent`/`slow`/`volatile`(運用値は mutable 側) | `lp_str` |
| 9 | `initial_tier` | `u8` | 著者の初期リスクティア。0=Tier-L / 1=Tier-H([設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.5) | 1バイト |

**過不足判断**:

- `prompt_hash` / `regurgitation_check`(Architecture.md §6 データモデルに記載)は Phase2 まで不採用とする(将来 `schema_ver` を上げて追加)。
- `network`(public/company/private)は core に含めない。ネットワークモードはノード属性であり、core に焼き込むと同一事実がモード違いだけで別 `entry_id` に重複してしまう。
- `initial_volatility_class` / `initial_tier` を署名対象に含める理由は、これらを「真として信頼させる」ためではなく、**著者の主張を固定して事後説明責任を成立させるため**である(スラッシング根拠、[Architecture.md](./Architecture.md) §10.3 攻撃A — 一次記載は[信頼性設計メモ.md](./信頼性設計メモ.md) §7 フェーズ3。「明らか毒」判定([設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §4.4)の基礎)。受信側は運用判断にこの値を一切信頼せず、必ず §6 手順8で再導出する。

## 3. 正準化の厳密仕様

**採用: 長さ接頭辞バイナリ(serde非依存)。代替: JCS(RFC 8785)。**

採用理由:

1. core は浮動小数点数をゼロ個しか持たず、JCS 最大の難所である数値正準化が不要になる。
2. 長さ接頭辞方式はエスケープも数値表現も曖昧性がゼロであり、C#/Rust 間のバイト一致が構造上自明になる。
3. serde を認証経路から完全に排除でき、[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §2.2 弱点2を設計目標として満たす。

JCS はコアには使わない。JCS の正準化ロジック自体が実装依存の再現性問題を持ち込みうるためであり、可読なデバッグ用途はハッシュ対象でないエンベロープJSON(§6)で十分に足りる。

### 厳密定義

```text
domain_tag := ASCII "nyllm/entry/v1\n"(15バイト固定(= "nyllm/entry/v1\n" の実バイト長)・長さ接頭辞なし)
u16(x) := 2バイト big-endian / u32(x) := 4バイト big-endian
lp_str(s) := u32(bytelen(nfc(s))) || nfc(s) の UTF-8 バイト列
nfc(s) := Unicode NFC 正規化(Rust: unicode-normalization / C#: string.Normalize(NFC))
```

### immutable_core 正準バイト列(フィールド固定順・ソートしない)

```text
domain_tag
u16(schema_ver)
lp_str(question_norm)
u32(facts.len()) + 各 fact: lp_str(s) lp_str(p) lp_str(o)
lp_str(provenance.agent)
lp_str(provenance.model)
lp_str(provenance.embedder_model_id)
lp_str(created)
lp_str(initial_volatility_class)
u8(initial_tier)
```

**規則**:

- キーソートは使用しない(固定順そのものが正準)。
- 文字列は必ず NFC → UTF-8(エスケープという概念自体が存在しない)。
- 数値は `u8`/`u16`/`u32` の big-endian 固定幅(LEB128は使わない)。浮動小数点数は core に存在しない。
- 配列は `u32` 個数接頭辞 + 要素列。
- 文字列内の改行等はバイトとしてそのまま扱う(長さ区切りのため区切り文字問題が発生しない)。
- C#/Rust で同一入力から同一バイト列が構造的に再現される。ゴールデンテスト(§9)で16進スナップショットを固定し、C#実装のクロス参照基準とする。

## 4. entry_id と question_key

```text
core_bytes  := immutable_core 正準バイト列
entry_id    := hex(sha256(core_bytes))              // 内容アドレス(回答インスタンスID)
qkey_bytes  := "nyllm/qkey/v1\n" || lp_str(fold(question))
question_key := hex(sha256(qkey_bytes))             // 検索・重複排除・版束ね
fold(q)     := NFC → 小文字化 → 前後trim → 連続空白を単一空白へ畳む
```

ハッシュ関数は SHA-256 で統一する。`entry_id` と `question_key` はドメイン分離により別タグ配下に置かれ(§11 ドメインタグ登録簿)、将来追加される witness/revocation 署名も必ず別タグを持つ。

`created` は `entry_id` に含める(同一事実の別版を別IDとして扱う。§5.2「複数版併存」)。**重複排除は `entry_id` ではなく `question_key` 側で行う**([設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §2.2 弱点4「`created` をハッシュに含めるため同一Q&Aが無限に重複エントリ化する」への解消)。

### question_key の「正規化質問クラスタ」

- **PoC暫定**: `fold(question)` の完全一致ハッシュ。言い換え(パラフレーズ)はクラスタされない(`MockEmbedder` の限界と同種の意図的制約)。完全重複排除のみを担う。
- **将来形(要注意)**: 「クラスタ代表ハッシュ」(意味的に近い質問群を1つの `question_key` にまとめる方式)は、ノード間で非一貫となり分散重複排除を壊す。推奨は `question_key` を content-based(`fold` 完全一致)のまま維持し、意味クラスタリングは検索層(embedding近傍検索)で担わせることである。**identity層(誰と誰が同一エントリを指すか)と search層(何が意味的に近いか)を混ぜない。**

## 5. 署名

対象: `author_sig = Signer::sign(core_bytes)`。`entry_id` と同一のバイト列に署名する(ハッシュと署名が同じ対象を覆うことで、[設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §2.2 弱点1「未署名フィールド経由の改竄が `verify()` を素通りする」を解消)。

**`Signer` trait 変更(必須)**: 現行の `sign_hex(&self, payload: &str)` はテキスト前提であるため、`sign_bytes(&self, &[u8]) -> String` / `verify(&self, pub_hex, sig_hex, &[u8]) -> bool` へ変更する(`core_bytes` はバイナリであり `str` に載らないため)。

**ドメイン分離**: `core_bytes` 先頭に `nyllm/entry/v1\n` を含むため、将来の署名(witness/revocation 等、別タグ)との cross-protocol 流用が構造的に不可能になる([設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §2.2 弱点6の解消)。

**`DummySigner` → `Ed25519Signer` の不変条件**: いずれの実装も `core_bytes` への署名を `verify` で再検証して一致することを保証する。改良案A(署名対象バイト列化)により `verify` は保存バイト列を直接ハッシュ・署名検証するようになるため、`Signer` 実装が変わっても不変条件(ハッシュ=改ざん検知/署名=偽造防止の分離、[信頼性設計メモ.md](./信頼性設計メモ.md) §4)を維持する。

**付随修正([設計レビュー_2026-07.md](./設計レビュー_2026-07.md) §2.2 弱点6)**: `DummySigner` の現行実装 `sha256(secret||payload)` は長さ拡張攻撃が成立しうるため、`HMAC-SHA256` へ変更する。

## 6. 保存レイアウトと load/verify 手順

```text
cache_store/
  <entry_id>.entry        ← 不変。serde JSON エンベロープ(authoritative は中の core_b64 のみ)
  <entry_id>.state.json   ← 可変。mutable_state(署名しない・各ノードが上書き)
```

`<entry_id>.entry` エンベロープ自体は非authoritative(serde自由)である。

```jsonc
{ "schema_ver": 1, "core_b64": "…", "author_pub": "…", "author_sig": "…" }
```

**要点**: serde を使うのは「ハッシュされないエンベロープ」と「state」のみである。`core_bytes` は base64 の不透明ブロブとしてエンベロープに格納される。`verify` は base64 復号後のバイト列を直接ハッシュ・検証するため、エンベロープJSONのキー順もエスケープも結果に一切影響しない。

### load/verify 手順(順序厳守)

1. `<id>.entry` を serde でパースし `core_b64` / `author_pub` / `author_sig` を取得する。
2. `core_bytes = base64_decode(core_b64)`。
3. `sha256_hex(core_bytes)` がファイル名 `<id>` と一致するか確認する。不一致なら drop(改ざん検知)。
4. `Signer::verify(author_pub, author_sig, core_bytes)` が真か確認する。偽なら drop(偽造防止)。
5. `parse_core(core_bytes)` で `ImmutableCore` を復元する。失敗すれば drop(形式不正)。
6. `question_key` を `core.question_norm` から再計算する(保存値があっても信頼しない)。
7. `embedding = local_embedder.encode(core.question_norm)` を再計算する(ファイルには非保存。改良案C)。
8. `judge_entry` を再実行し、`shareable` / `tier_operative` / `volatility_class_operative` を再導出する(送信者側の値は信頼しない)。
9. `<id>.state.json` が存在すれば `confidence`/`evidence`/`witness`/`anchor`/`stake`(Phase2は空)を読み込む。ただし `shareable`/`tier`/`volatility` の運用値は常に手順8の再導出結果を採用し、ディスク上の値で上書きしない。
10. 復元したエントリを `entries` に push する。

### embedding と embedder_model_id(改良案Cの正しい実装形)

`embedding` は純粋にローカルな索引データである。各ノードは自ノードの embedder で `question_norm` を再embeddingし、ファイルには保存しない。`core.provenance.embedder_model_id` は著者が使用した識別子であり、情報・分析用途(互換性ゲートとしては使わない)。ノードは `mutable_state.local_embedder_id` に自ノードが索引に使っている embedder を記録し、embedder を変更した場合は全件を再embeddingする(異なるベクトル空間の混入事故を防止するため)。

### 重複排除・バージョニング

register 時に `question_key` を計算する。同一 `entry_id` は完全重複としてスキップする。同一 `question_key` かつ異なる `entry_id` は別版として併存させる(§5.2)。

## 7. 現行 CacheEntry からの移行

`schema_ver = 1` を新形式に付与する。現行 `cache_store` は旧形式であり非互換となる。PoC であるため既存キャッシュは破棄でよい(`main.rs` は既定で `cache_store` を wipe し、`--keep` 指定時のみ既存ファイルを保持する挙動を維持する)。**旧形式から新形式への変換ロジックは書かない。** ロード時に `.entry` のパースに失敗した場合、または `schema_ver` が不一致の場合は黙って drop する(`--keep` 指定で旧形式ファイルが残存している場合は、起動時に非対応 `schema_ver` を検出したら警告ログを出した上で drop する)。

## 8. Rust型スケッチ(Allman・日本語コメント)

以下は設計ノートに含めるコードスケッチであり、実装時に poc-core-dev がこれを起点にする(CLAUDE.md の Allman ブレース・日本語コメント規約に準拠)。

```rust
// 不変コア(署名・ハッシュ対象。作成後不変。浮動小数を含まない)。
// 正準バイト列は encode_core() が serde 非依存で生成し、entry_id と author_sig は
// そのバイト列に対して計算する(改良案A)。
#[derive(Debug, Clone, PartialEq)]
pub struct ImmutableCore
{
    pub schema_ver: u16,
    pub question_norm: String,            // NFC正規化・trim済み
    pub facts: Vec<FactTriple>,           // triples.rs の型を再利用
    pub provenance: Provenance,
    pub created: String,                  // UTC RFC3339・秒精度・Z終端
    pub initial_volatility_class: String, // 著者の初期主張(運用値はMutableState側)
    pub initial_tier: Tier,               // 著者の初期リスク分類
}

// 出所メタ(agentを署名対象化=弱点1対応)。
#[derive(Debug, Clone, PartialEq)]
pub struct Provenance
{
    pub agent: String,
    pub model: String,               // モック時は ""
    pub embedder_model_id: String,   // 著者が使ったembedder識別子(情報用)
}

// リスクティア(§4.5 幻覚パリティ2ティア)。u8正準表現: Low=0 / High=1。
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum Tier
{
    Low,  // Tier-L: パリティ適用領域
    High, // Tier-H: パリティ禁止・別機構へ
}

// 可変状態(署名しない・entry_idに含めない・各ノードが導出/更新する)。
// 送信者の値は信頼せず、shareable/tier/volatility運用値はロード時に再導出する。
#[derive(Debug, Clone)]
pub struct MutableState
{
    pub volatility_class_operative: String,
    pub volatility_confidence: f32,
    pub volatility_evidence: Vec<String>,
    pub shareable: bool,
    pub share_reason: String,
    pub tier_operative: Tier,
    pub local_embedder_id: String,
    // --- Phase2 空スロット(型のみ確保。Phase1では None/空) ---
    pub trust: Option<Trust>,               // S4
    pub witness_sigs: Vec<WitnessSig>,      // S3(社内は共通時計で代替=空)
    pub anchor_proof: Option<AnchorProof>,  // S3(§4.2)
    pub stake: Option<Stake>,               // S4(§4.3)
}

// インメモリのエントリ(コア+コアバイト列+署名+可変状態+ローカルembedding)。
pub struct CacheEntry
{
    pub entry_id: String,     // hex(sha256(core_bytes))
    pub question_key: String, // hex(sha256(qkey_bytes))
    pub core: ImmutableCore,
    pub core_bytes: Vec<u8>,  // authoritative な正準バイト列(保存はbase64)
    pub author_pub: String,
    pub author_sig: String,
    pub state: MutableState,
    pub embedding: Vec<f32>,  // ローカル再計算・非保存(改良案C)
}

// ディスク上のエンベロープ(非authoritative=serde自由。preserve_order無関係)。
#[derive(Serialize, Deserialize)]
struct EntryEnvelope
{
    schema_ver: u16,
    core_b64: String,
    author_pub: String,
    author_sig: String,
}
```

**serde 構成の方針**: authoritative なハッシュ・署名は `encode_core()`(手書き・serde非依存)が担い、serde は `EntryEnvelope` と state JSON にのみ使う。よって `preserve_order` の可否はハッシュ安定性に一切影響しない(CLAUDE.md が定める「`preserve_order` を有効化しない」という禁止事項が構造的に無意味化する=より強い保証が得られる)。

Phase2型(`Trust`/`WitnessSig`/`AnchorProof`/`Stake`)はフィールドを持たない空 struct として今宣言し、中身は Phase2 で定義する(過剰設計回避。§4.7「動く的撃ち」の回避)。

## 9. テスト観点(test_cache 拡張。CLAUDE.md規則4)

| 観点 | 確認内容 |
|---|---|
| 正準決定性 | 同一 `ImmutableCore` を `encode_core` で2回エンコード → バイト完全一致 → `entry_id` 一致 |
| ゴールデン | 既知コアの `core_bytes` の16進スナップショットをピン留め(C#実装のクロス参照基準) |
| round-trip | `encode_core` → `parse_core` → 同一構造体。base64エンベロープ経由でも一致 |
| 改ざん検知(2経路分離) | (a) `question_norm` を1バイト書換 → `entry_id` 不一致で drop /(b) `author_sig` を書換 → 署名検証失敗で drop |
| 可変状態の非影響 | `state` の `confidence`/`shareable` を書換 → `entry_id` は不変(コアと分離されていることの証明) |
| 受信側再導出 | disk 上の `shareable=true` を偽装 → ロード後は再導出値が採用される |
| question_key 重複排除 | `fold` 同値の2質問 → 同一 `question_key`。異版(`created` 違い)は別 `entry_id` で併存 |
| NFC正規化 | 合成済み/分解済みの同一質問 → 同一 `question_key`/`entry_id` |
| Tier round-trip | `initial_tier = High` が正準 `u8` → パース → 同値。`tier_operative` の再導出も確認 |
| embedding非保存 | `.entry`/`.state.json` に `embedding` が含まれない。ロード時に再計算される |
| Ed25519 | `--features ed25519` で `core_bytes` への `verify` が通る(`sign_bytes` 経路) |
| no-float-in-core | `ImmutableCore` に `f32`/`f64` が存在しないこと(構造レビュー/型テスト) |

## 10. 要判断点(fable推奨を既定採用。各に採用/代替を明記)

1. `initial_tier`/`initial_volatility_class` を署名対象coreに入れるか → **採用(推奨)**。著者主張の固定=事後スラッシング根拠になる。受信側は信頼せず再導出する。代替=mutable側のみに置く(この場合スラッシング証拠が消える)。
2. `question_key` のクラスタ戦略 → **content-based維持(推奨)**。クラスタ代表ハッシュはノード間で非一貫となり分散重複排除を壊す。意味クラスタは検索層に担わせる。Public Phase2でも identity層は content-based のままとする。
3. 正準形 → **バイナリ(推奨)**。代替=JCS(可読だが serde 依存リスクを一部戻してしまう)。
4. `created` を `entry_id` に含めるか → **含める(推奨、版識別のため)**。代替=含めない(この場合、同一事実の再登録が同一IDの衝突として扱われ、§5.2「複数版併存」と矛盾する)。
5. `Signer` trait の bytes化 → **必須変更(推奨)**。既存呼び出し元(`main.rs`/テスト)へ影響が波及するため、実装時に明示的に扱う。
6. `author_sig`/`author_pub` の置き場 → **`.entry` エンベロープ内(推奨)**。代替=別 `.sig` ファイル。
7. ドメインタグ登録簿の新設 → **採用(推奨)**。`nyllm/entry/v1`・`nyllm/qkey/v1` に加え Phase2予約タグ(`nyllm/witness/v1`・`nyllm/revocation/v1`)を docs 側に1枚にまとめる(§11)。cross-protocol 流用を将来にわたって防ぐ。
8. Phase2型の空宣言粒度 → **空structのみ(推奨)**。`witness`/`anchor`/`stake` の中身は今書かない(過剰設計=動く的撃ち回避)。配置と署名境界だけを今確定する。

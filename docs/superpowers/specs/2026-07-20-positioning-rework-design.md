# Positioning 再設計 — うたい文句を「省力の物語」から「省力×信頼の両輪」へ

- 作成日: 2026-07-20
- ステータス: **設計完了・オーナー承認済み(2026-07-20)**。実装(README/docsへの適用)は本 spec レビュー後、project-leader 経由で design-docs-editor に割り当てる。
- 種別: プロジェクトの対外表現(positioning / うたい文句)の再定義。**コード・アーキテクチャ・S1〜S7の段階定義には一切触れない**。
- 前提ドキュメント: [README.md](../../../README.md) / [README.en.md](../../../README.en.md) / [AI_Concept.md](../../AI_Concept.md) / [信頼性設計メモ.md](../../信頼性設計メモ.md) / [Architecture.md](../../Architecture.md)

## 1. 背景と目的

### 発端
設計・実装を進めるうちに、プロジェクトの重心が **「分散セマンティックキャッシュ(=速い代替デバイス)」から「フィルタ/信頼層(=通す/止める/登録するの選別)」へ移った**、というオーナーの自己認識が出発点。ドライバでいうフィルタードライバ(既存スタックに透過的に割り込み、ショートサーキット=キャッシュヒット、選別ポリシー=共有ゲート/揮発性/内在信頼度)の比喩がよく当てはまる。

現行のうたい文句は主語が「キャッシュ」に寄っており(「AIに毎回考えさせるな。人類が一度出した答えを、まず探せ。」=省力化=探す、の一極のみ)、実態の重心(信頼層)を取りこぼしている。

### ゴール
positioning 全体(キャッチ・副題・「なぜ作るのか/何が新しいのか」の物語の主語)を、**省力(探す)と信頼(選別・検証)の両輪**で語り直す。フィルタ/選別のメタファーは公開コピーには**控えめに**(「信頼層」として一度だけ名詞で出し、以後は評判/独立検証/共有ゲートの具体語に落とす)出す。検閲・門番の含意を前面に出さない。

### 非ゴール
- コード・trait・しきい値・S1〜S7の段階定義/ゲート/進捗の変更(一切しない)。
- 設計判断そのものの変更(共有ゲートのデフォルト拒否、ネットワーク物理分離、3層評判等は既存のまま。本 spec は**それらを"どう語るか"だけ**を扱う)。
- ロゴ・ビジュアル・命名(NyLLM の名称は変えない)。

### オーナーの選択(ブレインストーミングで確定)
1. 射程: **positioning 全体**(キャッチ+副題+物語の主語)。
2. 中心: **両輪(省力×信頼)**。キャッシュという名詞と信頼層という名詞を対で立てる。
3. フィルタ語: **控えめに出す**(内部語彙に留めず一度だけ名詞提示、しかし主語にはしない)。

## 2. 採用する型(案B: 副題を"二層の製品"に格上げ)

検討した3案(A: 動詞だけ対にする最小手戻り / B: 副題を二層に格上げ / C: 信頼を入口に順序反転)のうち、**案B を採用**。理由:
- 射程「positioning 全体」に対し、動詞だけ差す案Aは物足りない。
- 「両輪」は *キャッシュ* と *信頼層* を対の名詞で立てるのが最も安定する。
- 副題の核 `分散セマンティックキャッシュ` を壊さないので、docs横断の改名コストを避けられる(名詞アイデンティティ保持)。
- フィルタ語も「信頼層」として一度出すだけで、「控えめに出す」に合致。
- 案Cは「両輪」よりも信頼側に振れ過ぎ(オーバーシュート)のため不採用。

## 3. 裏取り結果(2026年央時点・敵対的検証済み)

**結論: 「共有 × 分散 × 信頼層」の三点セットのセルは、2026年央時点で実際に空いている(確度: 高)。** 既存の製品・研究はすべて次の3バケツのいずれかに落ち、三つを同時に満たすものは見当たらない。

| バケツ | 具体例 | 欠けている軸 |
|---|---|---|
| (a) プライベート/単一テナントの意味キャッシュ | GPTCache, Redis LangCache(フルマネージドSaaS), Portkey/Kong/Cloudflare/Helicone ゲートウェイ, MeanCache(端末ローカル・共有するのは類似モデルのみ), waLLMartCache("multi-tenant"=むしろテナントを意味的に*分離*) | 共有していない |
| (b) "分散/P2P"を名乗るキャッシュ | LMCache, arXiv:2606.17059(P2P分散推論) | KV/prefixテンソルを*一運用者の内部で*共有するだけ。**答えの共有ではない**(トークン/テンソル層の遅延最適化) |
| (c) 信頼/評判・毒研究 | EigenTrust(P2P評判の古典, WWW 2003), NDSS 2026・Nature 2026(毒攻撃) | LLM共有キャッシュに*適用されていない* / 中央集権・単一システム前提 |

### positioning に効く3つの発見

1. **「なぜ薄いか=なぜこの設計か」が全部ソース付きで裏取れた**(構造的理由であって怠慢ではない):
   - **プライバシー/サイドチャネル**: Stanford ICML 2025 (arXiv:2502.07776) が、8社中7社がキャッシュをユーザー横断で共有しており、応答時間差から他人のプロンプトが推測できることを実証。→ NyLLM の **ネットワーク物理分離** の裏付け。
   - **鮮度(staleness)**: FreshCache (arXiv:2607.04281, 2026) は、安全に再利用できたのは全クエリの 15.6%(精度 95.5%)だけと定量化。→ **揮発性タグ+共有ゲートのデフォルト拒否** の裏付け。
   - **信頼の困難さ**: 下記2。

2. **⭐ 2026年に「意味キャッシュ毒」が現実の脅威として実証された。** NDSS 2026「When Cache Poisoning Meets LLM Systems」と Nature Scientific Reports (SAFE-CACHE, s41598-026-36721-w) が、埋め込み空間で意図的に衝突させ他人に攻撃者の答えを掴ませる攻撃を実証。既存の敵対的プロンプト防御は効かず「完全な緩和は依然として困難」。→ **NyLLM の信頼層は「必要」であることが第三者に証明され、かつ「まだ誰も解けていない」**。案B(信頼層を主語の片翼に立てる)の時勢的裏付け。

3. **EigenTrust という直系の設計図がある**(共有LLMキャッシュに適用された例はない)。3層評判の下敷きとして正当に引ける。

### ⚠️ 誠実さのためのガードレール(検証で棄却された主張=言ってはいけないこと)
- ❌ 「EigenTrust はシビルに強い(70%結託でも耐える)」→ 棄却(0-3)。評判 *計算* の設計図として引くのは可。**シビル耐性の証明としては引かない**。
- ❌ 「AWS/MS がすでに *組織横断の意味* キャッシュを共有している」→ 棄却。ただし *トークン単位* のユーザー横断キャッシュは実在(=上のサイドチャネル論文)。**「意味の答えの横断共有はまだ誰もやっていない」は言えるが、「一切のキャッシュ共有がない」とは言わない**こと。
- ⚠️ 本番ヒット率は 20〜45%(宣伝の 90〜95% は"精度"の話であって"ヒット頻度"ではない)。**過大な効率主張は避ける**。
- ⚠️ 「推論単価低下で割に合わない」という経済的理由は今回 *直接は* 裏取れず(プライバシー/鮮度/信頼の3つは取れた)。**経済論は控えめに**。
- ⏳ 時点性: 本スイープは 2026年央の点情報。毒/防御研究は活発に出ているため、公開前に再スイープが望ましい(§6)。

## 4. 成果物(適用する文言)

### 4.1 副題(名詞アイデンティティ)
> **NyLLM — 分散セマンティックキャッシュ、そして共有知の信頼層**

- `分散セマンティックキャッシュ` は残す(docs横断の改名コスト回避)。同格で「共有知の信頼層」を**一度だけ名詞で**足す=フィルタ語を控えめに出す。

### 4.2 キャッチ
**採用: B-1**
> **一度出した答えは、二度と考えない。ただし、信じる前に検証する。**

理由: 短く、両輪(省く/検証)が一文で立ち、毒研究の時勢とも噛む。

代替(トーン違い・不採用だが記録):
- B-2(時勢を突く): 「人類の答えを、まず探す。そして、汚された答えを通さない。」
- B-3(素直な両輪): 「AIに毎回考えさせるな。だが、共有された答えを無条件に信じるな。探す速さと、信じる根拠を、同じ層で。」

### 4.3 「何が新しいのか」書き出し(README 差し替え案)
> Semantic Cache 自体は既存技術(GPTCache, Redis LangCache)。**共有さえも、既にある**——ただしトークン単位で、しかも他人のプロンプトが時間差で漏れるという代償付きで(Stanford, 2025)。空白なのは「*意味の答え* を、*ユーザー/組織を跨いで*、*汚染に耐える信頼層とともに* 共有する」の三点セット。2026年、意味キャッシュへの毒注入は現実の攻撃として実証され、既存防御では防ぎ切れないことも示された(NDSS 2026)。つまり本プロジェクトが本丸に据える信頼層(3層評判・独立検証・共有ゲート)は、"あれば嬉しい" ではなく "共有を成立させるための前提" である。

## 6. 未解決事項 / 公開前の再確認

1. **再スイープ**: 公開直前に「共有×分散×意味の答え」キャッシュの新規参入(特に 2025-2026 の crypto×AI / 分散推論マーケット系スタートアップ)を再検索。今回のスイープでは確認されなかったが、この領域は動きが速く網羅していない。
2. **クロステナント境界の確定**: OpenAI/Azure/AWS/Alibaba が実際に横断共有しているのは prefix/KV か、意味(埋め込み一致)の答えまでか。「意味の答えの横断共有はまだ誰もやっていない」を断言するなら、この境界をもう一段裏取りしておくと安全。
3. **経済論の扱い**: 「推論単価低下+トークン単位キャッシュが簡単な勝ち筋を刈った」を主張に含めるなら、直接の証拠を1本取る(今回は未裏取り)。含めないなら現状の3理由(プライバシー/鮮度/信頼)で構成する。
4. **NDSS 2026 の "first" 表現**: 「初めて実証した」は同時期の並行研究があり争いがある(検証 2-1)。「意味キャッシュ毒は実証された脅威である」と書き、「◯◯が初」とは書かない。

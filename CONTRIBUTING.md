## 1. はじめに

**Status: 🚧 設計フェーズ完了 / PoC (S1: 単一ノード最小ループ) 進行中** — まだ動くネットワークはありません。

この段階のプロジェクトにとって、最も価値のある貢献は**議論**です。コードを書き始める前に、まず Issue / Discussion で構想を共有してください。大きな PR を予告なく投函されると、設計意図との突き合わせに時間がかかり、せっかくの作業が無駄になりかねません。

特に歓迎するテーマ:

- Embeddingモデル選定と共有用しきい値τの実測(0.9 は下限。§6 参照)

- 局所EigenTrustのパラメータ設計
- DHT実装の選定 (Kademlia系流用の可否)

もちろん、typo修正・ドキュメントの明確化・小さなバグ修正のような小規模な PR は、事前の議論なしでも歓迎します。

## 3. 開発環境

必要なのは [Rust](https://www.rust-lang.org/tools/install)(Cargo)のみです。既定構成の依存は小さな純Rustクレート(`serde` / `serde_json` / `sha2` / `hex` / `rand` / `chrono`)だけで、外部モデルやランタイムは不要です。

```sh
cd poc
cargo build
cargo run

# 既存キャッシュを保持したまま実行
cargo run -- --keep
```

Cargo feature(いずれも既定 off。**既定 off のままで最小ループが通しで動作します**):

| feature | ON にすると |
|---|---|
| `ed25519` | 署名が `DummySigner` から `ed25519-dalek` の実 Ed25519 署名に切り替わる |
| `onnx` | 実 Embedding のための**拡張点**。`OnnxEmbedder` は現状**未配線のスタブ**であり、動作する ONNX 経路ではない |

```sh
cargo build --features ed25519
```

**既定構成の制約を理解しておいてください**(いずれも意図的な割り切りで、デモ上も可視化されています):

- **既定の Embedder は `MockEmbedder`** — 文字 n-gram ハッシュです。**完全一致は sim=1.0 でヒットしますが、言い換え(パラフレーズ)はヒットしません。** 実際の意味的マッチングには実 Embedding モデルが必要です。
- **既定の Signer は `DummySigner`** — 鍵付き MAC であり、**公開検証可能ではないプレースホルダ**です。公開検証が要るなら `--features ed25519` を使ってください。

実行するとカレントディレクトリに `cache_store/`(エントリJSON)と `keys/node.key`(ノード鍵)が生成されます。**これらは git-ignore 済みの実行時生成物であり、コミットしないでください。**

詳細は [`poc/README.md`](./poc/README.md) を参照してください(ビルド手順の出典はそちらです)。

## 4. コーディング規約

- **コメントは日本語**で書いてください。
- **Allman ブレース** — 開き波括弧 `{` は必ず改行して行頭に置きます。

  ```rs
  fn abc()
  {
      // ...
  }
  ```

  **注意: `cargo fmt` をブレース整形の目的で一括実行しないでください。** ブレース位置を制御する rustfmt の `brace_style` オプションは nightly 限定(unstable)であり、stable ツールチェインで `cargo fmt` を走らせるとブレースが同じ行へ戻されてしまいます。Rust 側が nightly rustfmt を採用するまでは、Allman ブレースは手で維持する必要があります。C#/Blazor 側は `dotnet format` の既定が Allman なので、追加設定は不要です。
- **ファイル名の `nyllm_` prefix** — プロジェクト内部のファイルには `nyllm_` を付けます。使い捨ての調査スクリプトは `scripts/nyllm_probe*.py` に置いてください。
- **本実装のレイアウト** — 本番実装では、コア層は `src/core/`(Rust)、UI 層は `src/ui/`(Blazor / C#)に置きます。ただし **`poc/` は既存のフラット構成(`poc/src/*.rs`)を維持**します。このレイアウトを `poc/` に後付けしないでください。

## 5. テスト

コンポーネントを実装したら、**テストも必ず同時に実装してください**。`src/tests/test_<component>.rs` を作成し、テストが通ることを確認するまで作業を完了扱いにしないでください。

現状の実態にも触れておきます: PoC には `poc/src/tests/` に単体テスト一式(cache / volatility / signer、および `#[ignore]` 付きの検索ベンチ)があり、`cargo test` で実行できます。加えて `poc/src/main.rs` 自体が通しデモ(6問 → ヒット/ミス → 登録 → 改ざん検知)になっており、`cargo run` によるエンドツーエンドの動作確認手段としても引き続き有効です。テストを拡充する貢献も歓迎します。

## 6. 触る前に読むべき不変条件

以下はセキュリティモデルそのものです。**黙って変更すると設計が壊れます。** 根拠は [信頼性設計メモ](./docs/Winny_Type_Semantic_Cache_信頼性設計メモ.md) と [アーキテクチャ設計書](./docs/Winny_Type_Semantic_Cache_Architecture.md) にあります。

- **`entry_id = sha256(signed_payload)`** であり、これがディスク上のファイル名になります。`signed_payload()` は question + answer + created + volatility を対象とし、`serde_json::json!` で直列化されます(既定の `Map` の backing は `BTreeMap` なのでキーが整列され正準になります)。**`serde_json` の `preserve_order` cargo feature を有効化しないでください** — キー順が壊れて正準性が失われます。署名対象を変える場合は、id と `verify()` を**同時に**変更してください。
- **load 時検証** — `SemanticCache::load()` はハッシュを再計算し `author_sig` を検証します。どちらかに失敗したエントリは破棄されます。**ハッシュ = 改ざん検知(公開関数なので攻撃者も再計算できる) / 署名 = 詐称防止**であり、この2つを混同しないでください(設計メモ §4 参照)。なお `DummySigner::verify` が自ノード以外の鍵のエントリを拒否するのは **MAC の限界に由来する仕様であってバグではありません**。P2P の load 経路を書くときに `verify` を緩めると詐称防止が消えます。**解決策は `--features ed25519` であって `verify` の緩和ではありません。**
- **保守的な共有ゲート** — `share_gate()` は 文脈自立性 AND 事実型(非主観・非個人) AND 非揮発性 の AND 判定で、**既定は「共有しない」**です。「疑わしきは共有しない」に倒しています。人類規模では、悪いエントリ1件がネットワーク全体を汚染するためです。**PoC のゲートは L0(語彙ベース)のみの縮約**で、本来のゲートは L2「単独回答可」判定と regurgitation フィルタを含みます([Architecture §7](./docs/Winny_Type_Semantic_Cache_Architecture.md))。**AND 条件は追加してよいが、削ってはいけません。**
- **事実トリプルのみ保存(R1)** — `poc/src/cache.rs` の `answer: String`(平文回答)は **PoC の縮約**です。本来の規則は R1 = **事実トリプル+出典メタのみを保存し、回答は受信側で再合成、トリプル化できない内容は Public へ登録しない**([Architecture §11 R1 / T-H](./docs/Winny_Type_Semantic_Cache_Architecture.md))。PoC を共有方向に拡張する際、`answer` をそのままネットワークに載せないでください。
- **しきい値** — `LOCAL_THRESHOLD = 0.80` / `SHARED_THRESHOLD = 0.90`。共有側が厳しいのは、再現率より適合率を優先するためです。**共有用 τ ≥ 0.9 は要件(下限)**であり([Architecture §5.1 / T-A](./docs/Winny_Type_Semantic_Cache_Architecture.md))、単なるチューニング値ではありません。実測により**引き上げる**ことはできますが、**引き下げることはできません**。引き下げには先に Architecture §5.1 / T-A の変更が必要です。

> cache / signing / volatility / 共有ゲート に触れる PR は、脅威モデル観点でのレビュー対象になります。レビューに時間がかかることをあらかじめご了承ください。

## 7. PR の出し方

- 既定ブランチは `main` です。**`main` に直接コミットせず、ブランチを切ってください。**
- **1 PR 1 論点。** 無関係な変更を混ぜないでください。
- **設計判断を伴う変更は、先に Issue を立ててください。** 上記のとおり `docs/` が正なので、設計に影響する変更はまずドキュメント側で合意を取る必要があります。
- コミット前に `cargo build`・`cargo test`・`cargo run` が通ることを確認してください(§4 のとおり `cargo fmt` の一括実行には注意)。
